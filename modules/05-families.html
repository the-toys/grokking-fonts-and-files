Module 05

Identity & families

Where does the "bold" button get its glyphs? Not from a config file — from metadata baked into each font's name, OS/2 and head tables. This module dissects that metadata in real WenKai, then shows the exact trick build.py uses to weld two unrelated fonts into one family with a real bold — by editing four small tables and never moving a glyph.

concept

How separate files become one family

You think of "a font family" as one thing with several weights. On disk it is usually several independent files — one per weight/style. Nothing physically connects Regular.ttf and Bold.ttf: no manifest, no folder convention, no naming rule the OS enforces. So how does asking for bold reach into the right file?

The big idea A font family is not a config file. It is metadata baked into each file's name, OS/2 and head tables. CoreText (macOS), DirectWrite (Windows) and fontconfig/FreeType (Linux) all group files into one family when those files share a family name and advertise different weights/styles. Change the metadata and you change the family — without touching a single outline.

The name table: records keyed by four IDs

The name table is a bag of strings. Each record is keyed by a tuple (platformID, encodingID, languageID, nameID), so the same logical string can exist in many encodings and languages. The nameID says what the string means:

nameIDmeaningnotes
1Familylegacy family name (RIBBI grouping)
2Subfamilymust be one of Regular / Bold / Italic / Bold Italic — the "RIBBI" set
3Unique IDshould be globally unique
4Full namehuman-readable, e.g. "Family Bold"
6PostScript nameASCII, no spaces, must be globally unique
16 / 17Typographic family / subfamilyoverride 1/2 when present; used for families with >4 styles

The 1/2 pair can only express the four RIBBI styles, because 2 is restricted to that set. A family with Light, Medium, SemiBold… needs 16/17, which let the real names live free while 1/2 keep a legacy app happy by collapsing everything onto Regular/Bold. When 16/17 exist, they win. Remember that — it is the lever build.py pulls.

OS/2.usWeightClass: the number "bold" selects

Weight is a plain number in OS/2.usWeightClass: 400 = Regular, 700 = Bold (the scale runs 100–900). When an app "asks for bold" it is, under the hood, asking the family for the face whose weight class is 700. Get this number wrong and the bold button selects the wrong file — or nothing.

Style bits, stored twice — and cross-checked

The bold/italic facts are also stored as bit flags, redundantly, in two tables, and renderers cross-check the two. They must agree:

fieldbitvaluemeaning
OS/2.fsSelection00x01ITALIC
OS/2.fsSelection50x20BOLD
OS/2.fsSelection60x40REGULAR
head.macStyle00x01BOLD
head.macStyle10x02ITALIC

If fsSelection says BOLD but macStyle says not-bold, you have an inconsistent face: different platforms read different tables and disagree about what the file is.

PANOSE & the monospace flags

OS/2.panose is 10 bytes of coarse visual classification (serif style, weight, proportion…). The byte that matters here is index 3, bProportion: 9=Monospaced, 3=Modern (proportional). The other flag apps inspect for "is this monospace?" is post.isFixedPitch. Note both are advertisements — they describe the font, they do not make it monospaced (the advance widths do that, Module 4).

Faux bold is ugly If an app asks for bold and the family has no real bold face, many renderers synthesize one: they algorithmically smear (embolden) or slant (oblique) the Regular outlines. The result — "faux bold" / "faux italic" — is uneven and crude compared to a drawn bold. Shipping a real bold face is the whole point of getting the family metadata right.
walkthrough

WenKai's real identity, then the build.py retag

Here is WenKai's source identity, read straight out of LXGWWenKaiMonoGB-Regular.ttf:

fieldvaluemeaning
OS/2 version4the table layout / feature level
usWeightClass400Regular
fsSelection0b1000000bit 6 set = REGULAR (0x40)
head.macStyle0b0neither bold nor italic
achVendID"LXGW"4-byte vendor tag
PANOSE[2,2,5,9,0,0,0,0,0,0]byte0=2 Latin Text; byte3=bProportion=9=Monospaced

And the relevant Windows-platform (platformID 3) name records:

nameIDvalue
1 (Family)"LXGW WenKai Mono GB"
1 (Family, zh)"霞鹜文楷等宽 GB"
2 (Subfamily)"Regular"
4 (Full name)"LXGW WenKai Mono GB"
6 (PostScript)"LXGWWenKaiMonoGB-Regular"

Now the trick. We have two unrelated files: WenKai (a light, calligraphic Kai) and ZhenKai (a heavier Kai). We want one family "LXGW WenKai ZhenKai Mono GB" in which Regular = WenKai and Bold = ZhenKai. build.py's build_face() re-labels each file so the OS groups them. Step by step, with the real code:

1 · Same family, different subfamily

Both files are given the identical FAMILY string in nameID 1, but different nameID 2:

set_name(font, NAME_FAMILY, family)      # id 1 — SAME on both files
set_name(font, NAME_SUBFAMILY, style)    # id 2 — "Regular" vs "Bold"

That shared family name is the glue. set_name() writes each record for both the platforms that matter — Windows (3,1,0x409) and Mac (1,0,0) — after first calling removeNames() to clear stale records with the same ID, so no leftover original string lingers (this is why value must be ASCII: the Mac record can encode it).

2 · The weight number

os2.usWeightClass = weight   # 400 for WenKai, 700 for ZhenKai

This is literally what the bold button resolves against: ask FAMILY for weight 700 and you are handed ZhenKai.

3 · The style bits, in both places

build.py clears the three mutually-exclusive selection bits, then sets exactly one, and mirrors the bold bit into macStyle so the two tables agree:

os2.fsSelection &= ~(FS_ITALIC | FS_BOLD | FS_REGULAR)
os2.fsSelection |= FS_BOLD if bold else FS_REGULAR   # 0x20 or 0x40
head.macStyle  &= ~(MAC_BOLD | MAC_ITALIC)
if bold:
    head.macStyle |= MAC_BOLD                        # 0x01

So ZhenKai ends up with fsSelection bit 5 (BOLD, 0x20) and macStyle bit 0 (BOLD, 0x01); WenKai keeps fsSelection bit 6 (REGULAR, 0x40) and macStyle = 0. Coherent on every platform.

The payoff Any app that renders "LXGW WenKai ZhenKai Mono GB" and asks for bold is now handed ZhenKai — real, drawn bold strokes. No faux-bold smearing. No per-app config. This is the cross-platform version of the Linux fontconfig <alias> "treat zhenkai as wenkai bold" rule — done by editing the fonts instead of writing a rules file, so it works on macOS and Windows too.

4 · Advertise monospace without moving a glyph

post.isFixedPitch = 1 if monospace else 0
if getattr(os2, "panose", None) is not None:
    os2.panose.bProportion = 9 if monospace else 3   # 9=Mono, 3=Modern

This only flips the two flags a font picker filters on. It moves no advance width — and it doesn't need to, because the CJK glyphs are already full-width (advance == 1 em, Module 4). The build script even proves the equality at runtime, printing ZhenKai's 一 advance as 1000/1000 em = WenKai's — identical, so bold CJK columns align with regular and no re-metricizing is required.

5 · Unique IDs, and dropping 16/17

The PostScript and Unique-ID strings are derived from the family so every face/alias gets a distinct identity, and the typographic names are removed so the simple RIBBI grouping (1/2) wins:

ps_name = f"{family.replace(' ', '')}-{style}"   # e.g. LXGWWenKaiZhenKaiMonoGB-Bold
set_name(font, NAME_UNIQUE, ps_name)             # id 3 — break the dedup tie
set_name(font, NAME_PS, ps_name)                 # id 6 — ASCII, no spaces
font["name"].removeNames(nameID=NAME_TYPO_FAMILY)     # drop id 16
font["name"].removeNames(nameID=NAME_TYPO_SUBFAMILY)  # drop id 17
Why 3 and 6 must be unique — a real bug On macOS, CoreText keys faces partly on the Unique ID (3) and PostScript name (6). If our relabelled file kept the original's ID 3/6, CoreText could dedup or shadow ours against the still-installed original — and serve the wrong file for the family (a real font-priority bug). Each alias therefore gets its own unique 3/6. And the typographic names 16/17 are removed because if present they would override our 1/2 and re-split the two files apart, defeating the Regular+Bold grouping.

What it deliberately does NOT touch

Everything above lives in just four tables: name, OS/2, head, post. The heavy tables — glyf, cmap, hmtx — are left byte-identical. This is Module 1's lesson cashing in: tables are independent blobs, so you rewrite a few kilobytes of metadata and leave 24 MB of outlines untouched. It is also what makes the shared-table .ttc possible: because every Regular face has the exact same glyf bytes (and every Bold face likewise), the collection writer can store that outline data once — the setup for Module 7.

See it for yourself specimens/extract.py LXGWWenKaiMonoGB-Regular.ttf prints usWeightClass, fsSelection, macStyle, PANOSE and the name records shown above. Run build.py and watch it print the 一 advance equality before it writes LXGWHybrid.ttc.

5.1concept

Decode the selection bits. WenKai's OS/2.fsSelection is 0b1000000. Which bit is set, and what does it mean? What would 0b0100000 mean instead?

Show solution 0b1000000 = bit 6 set = REGULAR (0x40) — the value WenKai carries as the Regular face. 0b0100000 = bit 5 set = BOLD (0x20) — what build.py sets on ZhenKai. (Bit 0 = ITALIC, 0x01.)
5.2concept

Two places for one fact. Why does the format store the bold flag in both OS/2.fsSelection and head.macStyle? What is the risk if they disagree?

Show solution Historical layering: head.macStyle is the old Apple/TrueType style field; OS/2.fsSelection is the later Windows/OpenType one. Both survive, and different renderers/platforms read different ones. If they disagree the face is inconsistent — one platform treats it as bold, another as regular, so it may be selected for the wrong slot or get faux-styled. That is why build.py writes both coherently.
5.3concept

Family glue, in your own words. In 2–3 sentences, explain how two physically separate font files end up in the same "family" with a working bold — and name the specific metadata that does it.

Show solution Both files carry the same nameID 1 family string but different nameID 2 subfamilies ("Regular" vs "Bold") and different OS/2.usWeightClass (400 vs 700), with matching fsSelection/macStyle style bits. The OS's font system (CoreText / DirectWrite / fontconfig) groups files by family name and routes a request for "bold" to the face advertising weight 700 — no config file involved.

5.5concept

Why unique IDs? Why must nameID 3 (Unique ID) and nameID 6 (PostScript) differ across the aliases and against the originals? What bug appears on macOS if two faces share them?

Show solution CoreText partly identifies faces by ID 3/6. If two files share them, CoreText can dedup or shadow one against the other — treating them as the same face — and serve the wrong file when the family is requested (a font-priority / shadowing bug). Giving each alias a unique family-style PostScript/Unique ID keeps every face distinct, so the hybrid is served instead of the still-installed original.
5.6concept

Capstone-lite: the table ledger. List exactly which tables build.py modifies and which it deliberately leaves untouched. Why does leaving glyf/hmtx untouched preserve terminal column alignment (Module 4), and how does it enable the shared-table TTC (Module 7)?

Show solution Modified: name (ids 1,2,3,4,6; removes 16/17), OS/2 (usWeightClass, fsSelection, panose.bProportion), head (macStyle), post (isFixedPitch). Untouched: glyf, cmap, hmtx, loca, and the rest. Leaving hmtx/glyf alone means advance widths don't change — CJK stays full-width (advance == 1 em), so bold columns line up with regular in a terminal with no re-metricizing. And because every Regular face shares byte-identical glyf (every Bold face likewise), the .ttc writer can store that heavy data once via shareTables — the collection is ~the size of the two sources, not 9×. That's Module 7.
