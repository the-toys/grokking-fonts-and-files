Module 05

Identity & families

Click bold in any app and the glyphs change. Where do they come from? Not a config file — from metadata baked into each font's name, OS/2 and head tables. This module reads that metadata out of real WenKai, then uses it for a concrete goal: weld two unrelated fonts — WenKai and ZhenKai — into one family whose bold is a real, drawn second face, by editing four small tables and never moving a glyph.

Exercises are interleaved — each sits under the field it tests. Concept ones have a solution; programming ones have a collapsible ✓ Check your answer.

concept · 1

A family is metadata, not a config file

You think of "a font family" as one thing with several weights. On disk it is usually several independent files — one per weight/style. Nothing physically connects Regular.ttf and Bold.ttf: no manifest, no folder convention, no naming rule the OS enforces. So how does asking for bold reach into the right file?

The big idea A font family is not a config file. It is metadata baked into each file's name, OS/2 and head tables. CoreText (macOS), DirectWrite (Windows) and fontconfig/FreeType (Linux) all group files into one family when those files share a family name and advertise different weights/styles. Change the metadata and you change the family — without touching a single outline.

Why does any of this matter? Because if you get it wrong, the bold button still does something — something ugly:

The stakes: faux bold is ugly If an app asks for bold and the family has no real bold face, many renderers synthesize one: they algorithmically smear (embolden) or slant (oblique) the Regular outlines. The result — "faux bold" / "faux italic" — is uneven and crude compared to a drawn bold. Shipping a real bold face, and labelling it so the OS finds it, is the entire job of this module. Every field below exists to make the OS pick ZhenKai — a genuinely drawn heavier Kai — when you ask WenKai for bold.

So the plan is concrete: take two unrelated files, and edit just enough metadata that the OS sees one family with a real Regular and a real Bold. The next four sections introduce each field as the answer to one question — what groups the files? which one is bold? how is "bold" stored? how do pickers know it's monospace? — and then we do the retag for real.

concept · 2

The name table: what groups two files into one family

First question: what does the OS key on to decide two files belong together? The answer lives in the name table — a bag of strings. Each record is keyed by a tuple (platformID, encodingID, languageID, nameID), so the same logical string can exist in many encodings and languages. The nameID says what the string means:

nameIDmeaningnotes
1Familylegacy family name (RIBBI grouping)
2Subfamilymust be one of Regular / Bold / Italic / Bold Italic — the "RIBBI" set
3Unique IDshould be globally unique
4Full namehuman-readable, e.g. "Family Bold"
6PostScript nameASCII, no spaces, must be globally unique
16 / 17Typographic family / subfamilyoverride 1/2 when present; used for families with >4 styles

The mechanism that groups files is simple: same nameID 1 (Family), different nameID 2 (Subfamily). Give two files the identical family string and label one "Regular" and the other "Bold", and the OS files them under one family with two faces. That shared family name is the glue we'll use.

One catch worth banking now, because it bites us at the end. The 1/2 pair can only express the four RIBBI styles, because 2 is restricted to that set. A family with Light, Medium, SemiBold… needs 16/17, which let the real names live free while 1/2 keep a legacy app happy by collapsing everything onto Regular/Bold. When 16/17 exist, they win — so if either source font carries them, they'd override our 1/2 and re-split the two files apart. That makes "drop 16/17" a step in the merge, not an afterthought.

concept · 3

OS/2.usWeightClass: the number "bold" selects

Now the OS sees one family with two faces. Second question: when the user clicks bold, which of the two does it pick? By a number. Weight is a plain integer in OS/2.usWeightClass: 400 = Regular, 700 = Bold (the scale runs 100–900). When an app "asks for bold" it is, under the hood, asking the family for the face whose weight class is 700. Get this number wrong and the bold button selects the wrong file — or nothing.

5.3concept

Family glue, in your own words. You now know the two fields that do the grouping. In 2–3 sentences, explain how two physically separate font files end up in the same "family" with a working bold — and name the specific metadata that does it.

Show solution Both files carry the same nameID 1 family string but different nameID 2 subfamilies ("Regular" vs "Bold") and different OS/2.usWeightClass (400 vs 700), with matching fsSelection/macStyle style bits. The OS's font system (CoreText / DirectWrite / fontconfig) groups files by family name and routes a request for "bold" to the face advertising weight 700 — no config file involved.
concept · 4

Style bits, stored twice — and cross-checked

The subfamily name and weight number aren't the only places "this is bold" is recorded. The bold/italic facts are also stored as bit flags, redundantly, in two tables, and renderers cross-check the two. They must agree:

fieldbitvaluemeaning
OS/2.fsSelection00x01ITALIC
OS/2.fsSelection50x20BOLD
OS/2.fsSelection60x40REGULAR
head.macStyle00x01BOLD
head.macStyle10x02ITALIC

If fsSelection says BOLD but macStyle says not-bold, you have an inconsistent face: different platforms read different tables and disagree about what the file is. So our retag will have to write both, coherently.

5.1concept

Decode the selection bits. WenKai's OS/2.fsSelection is 0b1000000. Which bit is set, and what does it mean? What would 0b0100000 mean instead?

Show solution 0b1000000 = bit 6 set = REGULAR (0x40) — the value WenKai carries as the Regular face. 0b0100000 = bit 5 set = BOLD (0x20) — what we set on ZhenKai in the merge. (Bit 0 = ITALIC, 0x01.)
5.2concept

Two places for one fact. Why does the format store the bold flag in both OS/2.fsSelection and head.macStyle? What is the risk if they disagree?

Show solution Historical layering: head.macStyle is the old Apple/TrueType style field; OS/2.fsSelection is the later Windows/OpenType one. Both survive, and different renderers/platforms read different ones. If they disagree the face is inconsistent — one platform treats it as bold, another as regular, so it may be selected for the wrong slot or get faux-styled. That is why the merge writes both coherently.
concept · 5

PANOSE & the monospace flags

One last piece of identity, because this is a monospace family and font pickers let you filter for that. A picker doesn't measure glyphs — it reads an advertisement. OS/2.panose is 10 bytes of coarse visual classification (serif style, weight, proportion…). The byte that matters here is index 3, bProportion: 9=Monospaced, 3=Modern (proportional). The other flag apps inspect for "is this monospace?" is post.isFixedPitch. Note both are advertisements — they describe the font, they do not make it monospaced (the advance widths do that, Module 4). The retag must set them so the hybrid still shows up under a monospace filter.

5.4programming

Read the identity yourself. Before we change anything, confirm the starting state. Using fontTools, open WenKai and print usWeightClass, bin(fsSelection), bin(head.macStyle), and name IDs 1/2/6. Match them against the two tables above.

# python this.py LXGWWenKaiMonoGB-Regular.ttf
import sys
from fontTools.ttLib import TTFont
f = TTFont(sys.argv[1])
os2, head, name = f["OS/2"], f["head"], f["name"]
print("usWeightClass", os2.usWeightClass)
print("fsSelection  ", bin(os2.fsSelection))
print("macStyle     ", bin(head.macStyle))
for i in (1, 2, 6):
    print("name", i, repr(name.getDebugName(i)))

Paste your output — I'll confirm usWeightClass=400, fsSelection=0b1000000, macStyle=0b0, and the three name strings against the table.

Now the retag. We have two unrelated files: WenKai (a light, calligraphic Kai) and ZhenKai (a heavier Kai). We want one family "LXGW WenKai ZhenKai Mono GB" in which Regular = WenKai and Bold = ZhenKai. We'll write a small routine — call it build_face() — that re-labels each file so the OS groups them. Each step writes one field you just met:

1 · Same family, different subfamily (concept 2)

Both files are given the identical FAMILY string in nameID 1, but different nameID 2:

set_name(font, NAME_FAMILY, family)      # id 1 — SAME on both files
set_name(font, NAME_SUBFAMILY, style)    # id 2 — "Regular" vs "Bold"

That shared family name is the glue. set_name() writes each record for both the platforms that matter — Windows (3,1,0x409) and Mac (1,0,0) — after first calling removeNames() to clear stale records with the same ID, so no leftover original string lingers (this is why value must be ASCII: the Mac record can encode it).

2 · The weight number (concept 3)

os2.usWeightClass = weight   # 400 for WenKai, 700 for ZhenKai

This is literally what the bold button resolves against: ask FAMILY for weight 700 and you are handed ZhenKai.

3 · The style bits, in both places (concept 4)

We clear the three mutually-exclusive selection bits, then set exactly one, and mirror the bold bit into macStyle so the two tables agree:

os2.fsSelection &= ~(FS_ITALIC | FS_BOLD | FS_REGULAR)
os2.fsSelection |= FS_BOLD if bold else FS_REGULAR   # 0x20 or 0x40
head.macStyle  &= ~(MAC_BOLD | MAC_ITALIC)
if bold:
    head.macStyle |= MAC_BOLD                        # 0x01

So ZhenKai ends up with fsSelection bit 5 (BOLD, 0x20) and macStyle bit 0 (BOLD, 0x01); WenKai keeps fsSelection bit 6 (REGULAR, 0x40) and macStyle = 0. Coherent on every platform.

The payoff Any app that renders "LXGW WenKai ZhenKai Mono GB" and asks for bold is now handed ZhenKai — real, drawn bold strokes. No faux-bold smearing. No per-app config. This is the cross-platform version of the Linux fontconfig <alias> "treat zhenkai as wenkai bold" rule — done by editing the fonts instead of writing a rules file, so it works on macOS and Windows too.

4 · Advertise monospace without moving a glyph (concept 5)

post.isFixedPitch = 1 if monospace else 0
if getattr(os2, "panose", None) is not None:
    os2.panose.bProportion = 9 if monospace else 3   # 9=Mono, 3=Modern

This only flips the two flags a font picker filters on. It moves no advance width — and it doesn't need to, because the CJK glyphs are already full-width (advance == 1 em, Module 4). We already proved that equality in Module 4: ZhenKai advances 一 at 1000/1000 em = WenKai's — identical, so bold CJK columns align with regular and no re-metricizing is required.

5 · Give each face a unique name — 3 & 6 so the hybrid can't collide with the originals

ps_name = f"{family.replace(' ', '')}-{style}"   # e.g. LXGWWenKaiZhenKaiMonoGB-Bold
set_name(font, NAME_UNIQUE, ps_name)             # id 3 — Unique ID
set_name(font, NAME_PS, ps_name)                 # id 6 — PostScript name (ASCII, no spaces)

nameID 6 (PostScript name) and 3 (Unique ID) are meant to be globally unique — macOS CoreText uses them as a font's identity key. You almost certainly still have the original WenKai and ZhenKai installed, so if our relabelled Regular member kept WenKai's original name "LXGWWenKaiMonoGB-Regular", two files would claim the same identity. CoreText keeps only one font per PostScript name, so it deduplicates — and may keep the original, not ours. A request for our family then silently resolves to the wrong file: the hybrid doesn't render, or the Regular↔Bold link breaks because the member it needed got shadowed. Deriving each name from the new family gives every face a distinct identity that coexists with the originals.

6 · Drop the typographic names — 16 & 17 so the simple RIBBI names win

font["name"].removeNames(nameID=NAME_TYPO_FAMILY)     # drop id 16
font["name"].removeNames(nameID=NAME_TYPO_SUBFAMILY)  # drop id 17

Recall from concept 2 that 16/17 override 1/2 when present. If either source font still carries them, the OS reads each file's original typographic family — splitting the two files right back into their separate families and undoing the Regular+Bold grouping we just built. Stripping 16/17 forces the OS onto our simple RIBBI (1/2) names.

5.5concept

Why unique IDs? Why must nameID 3 (Unique ID) and nameID 6 (PostScript) differ across the aliases and against the originals? What bug appears on macOS if two faces share them?

Show solution CoreText partly identifies faces by ID 3/6. If two files share them, CoreText can dedup or shadow one against the other — treating them as the same face — and serve the wrong file when the family is requested (a font-priority / shadowing bug). Giving each alias a unique family-style PostScript/Unique ID keeps every face distinct, so the hybrid is served instead of the still-installed original.
concept · 7

What the retag deliberately does NOT touch

Everything above lives in just four tables: name, OS/2, head, post. The heavy tables — glyf, cmap, hmtx — are left byte-identical. This is Module 1's lesson cashing in: tables are independent blobs, so you rewrite a few kilobytes of metadata and leave 24 MB of outlines untouched. It is also what makes the shared-table .ttc possible: because every Regular face has the exact same glyf bytes (and every Bold face likewise), the collection writer can store that outline data once — the setup for Module 7.

See it for yourself specimens/extract.py LXGWWenKaiMonoGB-Regular.ttf prints usWeightClass, fsSelection, macStyle, PANOSE and the name records shown above — the inputs to the retag. Apply the six steps above to a copy of each file and you've assembled the family yourself; Module 7 then packs the relabelled faces into a single .ttc.
5.6concept

Capstone-lite: the table ledger. List exactly which tables the merge modifies and which it deliberately leaves untouched. Why does leaving glyf/hmtx untouched preserve terminal column alignment (Module 4), and how does it enable the shared-table TTC (Module 7)?

Show solution Modified: name (ids 1,2,3,4,6; removes 16/17), OS/2 (usWeightClass, fsSelection, panose.bProportion), head (macStyle), post (isFixedPitch). Untouched: glyf, cmap, hmtx, loca, and the rest. Leaving hmtx/glyf alone means advance widths don't change — CJK stays full-width (advance == 1 em), so bold columns line up with regular in a terminal with no re-metricizing. And because every Regular face shares byte-identical glyf (every Bold face likewise), the .ttc writer can store that heavy data once via shareTables — the collection is ~the size of the two sources, not 9×. That's Module 7.
