模块 02

cmap 与字形 ID

字体并不认识“字母”。它内部只认 字形 ID——0 … numGlyphs−1 范围内的小整数。cmap 表就是 从你文本中的 Unicode 码位通往这些数字的桥梁。它是文本变成像素的第一步， 而且刻意只做第一步。

练习穿插在正文里——读到哪道做哪道，做完把你的 答案/输出贴给 AI 导师。提示与自查答案都可以折叠展开。

概念

一个字符如何变成字形

字体能画的每样东西都由一个字形 ID（常写作“GID”）标识：一个 0 到 numGlyphs−1 之间的整数。这款字体的 numGlyphs 是 46830，所以合法的 GID 是 0 … 46829。对任何 sfnt 字体，有两条铁律：

  • 字形 0 永远是 .notdef——“缺字字形”，通常画成 一个空框或带叉的方框（俗称豆腐块）。它的存在保证了“没有字形”时也有东西可画。
  • 传给字体的文本是 Unicode 码位（U+0041、U+4E00……）， 不是字形 ID。总得有个东西负责翻译，这个东西就是 cmap。

所以渲染器拿到 "A一" 后干的第一件事，就是把每个码位拿去查 cmap，换成字形 ID，之后的每个阶段（轮廓、 度量、定位）用的都是这些 ID。心智模型如下，用的是这款字体真实的字形 ID：

"A"  → U+0041 ──cmap──▶ GID 1     ──glyf──▶ outline ──hmtx──▶ width
"一" → U+4E00 ──cmap──▶ GID 3171  ──glyf──▶ outline ──hmtx──▶ width
可迁移的洞见 cmap 是一次纯查表：码位 → 字形 ID，一进一出，没有 上下文。它看不到相邻的字符。所以一切依赖上下文的事——把 f+i 合成 fi 连字、挑选位置变体、换上 风格替代字形——都发生在更后面的整形引擎里，由 GSUB/GPOS 完成（模块 6）。把这条边界记牢：cmap 回答“这个字符是哪个字形”， 整形回答“结合前后这一串文本，实际该用哪些字形、摆在哪里”。
2.1概念题

那个特殊的字形。为什么字形 ID 0 特殊？当某个码位在 cmap 里没有条目时，屏幕上会出现什么？

看答案 每个 sfnt 字体都把字形 ID 0 保留给 .notdef——钦定的“缺字字形”。 当你请求一个字体没有映射的码位时，查表得到 GID 0（或者查不到条目，按 GID 0 处理），渲染器于是画出 .notdef：通常是一个空矩形或带叉的矩形，也就是 豆腐块。它传达的信息是“这款字体没有那个字符的字形”——文本和 编码都没问题，缺的是字符覆盖。
2.2概念题

cmap 到哪儿为止，整形从哪儿开始。比较预组合的单个字符 ﬁ（U+FB01，“fi”连字）和依次输入 f + i 两个字符。哪一个由 cmap 解决，哪一个需要 整形引擎？说明这条边界。

看答案 U+FB01 是一个真实的码位，所以 cmap 直接把它映射到那个 fi 连字字形——一次查表，完事。而先打 f 再打 i 得到的是 两个码位（U+0066、U+0069）；cmap 把它们各自独立地映射到 各自的字形，因为 cmap 根本没有“这俩挨着”的概念。 把它们合成一个连字字形是一个依赖上下文的决定，这是整形阶段 GSUB（liga 特性）的活儿。边界就在这里： cmap = 无上下文的字符→字形；整形 = 依赖上下文的字形→字形（可多个）。 模块 6 会展开讲。
概念

一张表，多张子表

cmap 表不是单一的映射，而是一列子表，每张 子表声明自己面向的受众和使用的编码。原因是历史性的：不同平台想要不同的 字符编码，OpenType 的选择是把几套映射打包进同一张表，而不是钦点 一个赢家。每张子表声明：

  • 一对 (platformID, encodingID)——这张子表给谁用；
  • 一个 format——码位→GID 的数据怎么编码。

渲染器扫一遍子表，从自己看得懂的里面挑最好的一张。

platformID 与 encodingID

platformID平台常见 (plat, enc) 组合
0Unicode(0,4) / (0,6) = 完整 Unicode
1Macintosh（历史遗留）(1,0) = Mac Roman
3Windows(3,1) = BMP，(3,10) = 完整 Unicode

这里面没有 Linux 的平台 ID——这个枚举比 Linux 桌面文本出现得还早。Linux （FreeType/HarfBuzz/fontconfig）、macOS（CoreText）和 Windows（DirectWrite）其实都只是读一张 Unicode cmap，而且实践中全都收敛到了 Windows 平台的 (3,10) 子表。这里的“Windows”实际上就是“大家都在用的那个标准”。

子表格式

format覆盖范围是什么
0256 个码位迷你的历史遗留字节表（每字符一个字节）
4BMP，U+0000…U+FFFF分段式 16 位映射
6一段连续的 16 位区间“裁剪数组”，一整段稠密映射
12完整 32 位，全部 17 个平面分段式 32 位映射
14变体选择符Unicode 变体序列（Variation Sequences）

今天真正要紧的是两种：format 4（基本多文种平面——头 65,536 个码位——的主力）和 format 12（把同一套 分段思路扩展到完整的 32 位空间，从而够得着 U+FFFF 之上的 补充平面）。format 0 和 6 是覆盖面很窄的历史遗留编码；format 14 根本不映射 字符——它处理的是“用那个字符的这个变体”这类序列。

2.3概念题

读懂子表的标签。某张子表声明 (platformID=3, encodingID=10, format=12)。说出它面向谁、能寻址的码位范围 是什么。

看答案 platformID 3 = Windows；encodingID 10 = 完整 Unicode。format 12 是 分段式 32 位格式，因此能映射整个 Unicode 空间—— 从 U+0000 到 U+10FFFF，全部 17 个平面——而不止 16 位的 BMP。在这款 字体里，这张子表映射了 46510 个码位。
解读

文楷的真实 cmap

下面是霞鹜文楷等宽 GB（LXGW WenKai Mono GB）——LXGWWenKaiMonoGB-Regular.ttf——货真价实的 cmap：五张子表。 nMappings 是每张子表映射的码位数量。

platformIDplatEncIDformatnMappings是什么
03443498Unicode 平台，仅 BMP（16 位）
041246510Unicode 平台，完整范围（32 位）
05140Unicode 变体序列（此处为空）
31443498Windows，BMP（16 位）
3101246510Windows，完整 Unicode（32 位）

逐行看：

  • (0,3) format 4 和 (3,1) format 4 都映射 43498 个码位——这款字体拥有的、落在 BMP （U+0000…U+FFFF）内的全部字符。一张标给 Unicode 平台，一张标给 Windows——而且 后面逐字节细读时会看到，它们字面意义上指向同一段字节。
  • (0,4) format 12 和 (3,10) format 12 都映射 46510 个码位——BMP 字符加上 U+FFFF 之外的 所有字符（同样，一份共享字节挂着两个标签）。
  • (0,5) format 14 映射 0 个：变体序列子表 存在，但不带任何序列，纯粹是个结构上的占位符。
可迁移的洞见 差值 46510 − 43498 = 3012，就是这款字体支持的、位于 16 位 BMP 之上的字符数量。在这款字体里，它们是散落的特殊字符——注音修饰符、 古汉语标记、小假名——范围从 U+107A5 到 U+FFFFD。 （有点出人意料：作为一款 CJK 字体，它并不包含 U+20000+ 的 CJK 扩展 B——覆盖范围永远要查证，不要想当然。）format 4 子表 在物理上不可能寻址其中任何一个：它的 码位是 16 位的。这就是字体要同时带一张 format 4 和一张 format 12 子表的全部原因。（另外注意 numGlyphs = 46830 > 已映射的 46510 个码位——有些 字形没有任何码位指向它，比如 .notdef 和只在整形时用到的替代字形。）

2.4概念题

为什么两种格式都要带？这款字体带了一张 format 4 子表（43498 条映射）和一张 format 12 子表（46510 条映射），看上去大量冗余。为什么两张都留着？具体地说： 如果删掉 format 12 子表、只留 format 4，会坏掉什么？

看答案 format 4 用 16 位码位，所以只能寻址 BMP（≤ U+FFFF）。U+FFFF 之上的 46510 − 43498 = 3012 个字符——这款字体携带的补充平面 特殊字符，比如 U+16FE3（一个古汉语标记）——将失去 映射。请求它们会得到 .notdef，渲染成豆腐块， 尽管字形明明就躺在字体里。正是 format 12 子表让这些字形 变得可达。

用 fontTools 查看字体

接下来两道练习要动手，所以先给出最小工具集——fontTools，读写字体的标准 Python 库（模块 1 的 sfnt 解析它替你做了）：

from fontTools.ttLib import TTFont
font = TTFont(path)              # open & parse the font
font["cmap"].tables           # list of subtables; each has .format, .platformID, .platEncID
sub.cmap                        # one subtable's {codepoint: glyphName} dict
font.getBestCmap()              # merged best-Unicode {codepoint: glyphName}
font.getGlyphID(name)           # glyph name -> integer GID

用 python 跑就行。接下来两道练习需要的就这些；后面的 工具一节还会再往下挖一点。

2.5编程题

数一数只有 format 12 够得着的码位。统计出现在 format 12 子表、但不在 format 4 子表中的码位数量——也就是 cp > 0xFFFF 的那些。与上面两个 nMappings 的差值对比。

提示 从 font["cmap"].tables 拿到原始子表。挑一张 .format == 4 和一张 .format == 12；每张都有一个以码位为键的 .cmap 字典。然后 做个集合差：len(set(fmt12.cmap) - set(fmt4.cmap))。预期 ≈ 46510 − 43498。
✓ 核对答案 3012。设 t4/t12 为 font['cmap'].tables 里的 format 4 和 format 12 子表：len(set(t12.cmap) - set(t4.cmap))，或者等价地 sum(1 for cp in t12.cmap if cp > 0xFFFF)。两者都等于 46510 − 43498 = 3012。

2.6编程题

把码位映射到字形名。用 fontTools 加载字体，调用 getBestCmap()，打印 A（U+0041）、一 （U+4E00），以及一个这款字体确实映射了的补充平面字符——例如 𖿣 （U+16FE3，一个古汉语标记）——的字形名。打印码位、字符和解析出的名字。然后是 真正的问题：仅凭返回的字典，你能对用了哪张子表下什么 结论——又有什么是你不能下的？（预警：一个“显而易见”的选择，比如 𠀀 U+20000，会返回 None——这款 GB 字体不覆盖它。要验证，别想当然。）

卡住了？给个方向（不给完整答案） getBestCmap() 返回一个普通的 {codepoint: glyphName} 字典，所以每次查询 就是 cmap.get(cp)。只要出现任何 cp > 0xFFFF，格式就暴露了 （format 4 物理上装不下它们）。但在断言是哪一张之前，先数一数 [(t.platformID,t.platEncID,t.format) for t in font["cmap"].tables] 里 有几张 format 12 子表——再试试 best is font["cmap"].tables[i].cmap，看对象同一性 帮不帮得上忙。
✓ 核对答案 A → "A"（gid 1）；一 → "u4E00"（gid 3171）；𖿣 U+16FE3 → "oldchineseiterationmodhan"（gid 21286）。
字典能告诉你的：这份映射是 format 12——它包含 0xFFFF 之上的码位（这里最大是 U+FFFFD），format 4 装不下， 空的 format 14 也提供不了。
字典不能告诉你的：是哪一张 format 12 子表。一共有两张， (0,4) 和 (3,10)，内容完全相同——而且因为它们在磁盘上共享同一个 offset，fontTools 把它们去重成了同一个字典对象，于是连 best is tables[(0,4)].cmap 和 best is tables[(3,10)].cmap 都 双双为 True。这个结果是真正无法区分的。答案是 (3,10)，仅仅因为它在 fontTools 文档写明的偏好顺序 cmapPreferences = (3,10),(0,6),(0,4),(3,1),… 里排第一——这是关于算法的事实，不是 关于数据的。教训：字典只暴露格式，永远暴露不了平台对。

自查见上——或把你的代码和输出贴给 AI 导师，让它核对。

二进制细读

第一步：cmap 自己也是个容器

在解读任何 format 4 的字节之前，你得先找到它们。从模块 1 的表目录我们知道 cmap 表位于文件偏移 187808。但这张表开头并不是子表数据——而是 它自己的小目录：一个 4 字节的头部 （version、numTables），后面每张子表跟一条 8 字节的编码记录： (platformID, encodingID, offset)，其中 offset 是 从 cmap 表的起点量起的。这又是模块 1 的老套路，只是往里多嵌套了一层。

2.7编程题

从原始字节解析 cmap 索引。用文件偏移 187808（来自模块 1）， 读出 cmap 表的 version 和 numTables，再逐条读编码记录。 打印每条 (platformID, encodingID, offset)。哪些记录共享同一个 offset？ 然后算出 (3,1) 子表的文件偏移，并确认那里开头两个字节解码出来 是 format 4。

卡住了？给个方向（不给完整答案） 以二进制方式读文件。cmap 头部是两个大端序 uint16 （">HH"）；每条记录是 uint16, uint16, uint32（">HHI"，8 字节）。记录里的 offset 相对于 cmap 表起点，所以子表的文件偏移是 187808 + recordOffset。到那里读 2 个字节，按 uint16 解包就是 format。（187808 可以自己按模块 1 的方式解析出来，也可以直接取 TTFont(path).reader.tables['cmap'].offset。）
✓ 核对答案 version=0, numTables=5。记录：(0,3)→44、(0,4)→50156、 (0,5)→217080、(3,1)→44、(3,10)→50156。所以 (0,3) 和 (3,1) 共享 offset 44（(0,4) 和 (3,10) 共享 50156）。(3,1) 子表位于文件偏移 187808 + 44 = 187852，那里的第一个 uint16 是 4——一张 format 4 子表。

每张 format 4 子表必须以一个 endCode 为 0xFFFF 的分段收尾——正是这个哨兵保证“找第一个 endCode ≥ c”的查找 一定会终止。

查表算法

  1. 在 endCode 里二分查找第一个满足 endCode[i] ≥ c 的分段 i。（三件套就是在这一步挣饭吃的。）
  2. 如果 c < startCode[i]，码位落在缝隙里 → 字形 0（.notdef）。
  3. 如果 idRangeOffset[i] == 0（简单路径）： gid = (c + idDelta[i]) & 0xFFFF。整个分段是一段连续的字形区间。
  4. 否则（指针路径）：字形 ID 住在 glyphIdArray 里，而 idRangeOffset[i] 是一个从它自己所在槽位量起的字节偏移，指进那个 数组。然后再加 idDelta。

两次完整的手算查表（都验证过）

简单路径——一（U+4E00）。它落在一个 idDelta = −16797、idRangeOffset = 0 的分段里：

gid = (0x4E00 + (−16797)) & 0xFFFF
    = (19968 − 16797) & 0xFFFF
    = 3171                       → glyph "u4E00"  ✓

（数字同理：'0' U+0030 在分段 1 → (48 + 210) = 258。）带符号的 idDelta 说白了就是“这一段码位映射到 平移了一个常数的一段字形 ID”——又省又密。

指针路径——A（U+0041）。它在分段 2 （startCode=0x3a、idDelta=0、idRangeOffset=3092）。规范给的 公式算出的是子表内部的一个字节地址：

addr = idRangeOffset[i]  +  2·(c − startCode[i])  +  &idRangeOffset[i]
     = 3092              +  2·(65 − 58)           +  9212
     = 12318
glyphIdArray @ 12318 = 1 ,  + idDelta(0)         = gid 1   → glyph "A"  ✓
(cross-checked: font.getGlyphID('A') == 1)
“从自己槽位量起的偏移”这个怪胎是怎么来的 idRangeOffset 是整个 OpenType 里最让人犯迷糊的东西，它是 C 指针运算的化石。在最初的 C 代码里，glyphIdArray 在内存中紧跟 在 idRangeOffset 后面，于是 *(&idRangeOffset[i] + idRangeOffset[i]/2 + (c − startCode[i])) 顺理成章地就 伸了进去。把偏移编码成相对于数组槽位自身，让 90 年代的代码一次 指针加法就能到位，连基址都不用另存。这份小聪明的账我们一直付到今天，付的是困惑—— 这也正是下一个格式把它扔掉的原因。

为什么长成这样——编码器的视角

一旦你换成写出这张表的工具的思路，这个格式就不再显得莫名其妙。 编码器手里有一堆（码位 → 字形 ID）对，想要最小的编码。它把码位分组成 连续的分段，每个分段在两种表示里挑便宜的那种：

区间内的码位对应的字形 ID选用的编码
连续也连续（c + k）只用 idDelta——存一个常数 k，没有逐字形数据
连续散乱glyphIdArray——把 ID 逐个列出来；idRangeOffset 指向这份列表
内部有洞—glyphIdArray，未映射的码位填 0

所以 idDelta 是常见情形（一整块字形按码位顺序排布）的压缩捷径， glyphIdArray 是 ID 对不齐时的 兜底。一个分段不能两头都占，所以才有两套机制。

连成一条的带子（为什么 idRangeOffset 是“从自己槽位量起”）

idRangeOffset[] 和 glyphIdArray[] 是一整段连续的 uint16 单元——glyphIdArray 只是它的尾巴。每个非零的 idRangeOffset 单元都是一支射向尾巴的箭，从它自己量起。 一个玩具分段，把 A,B,C 映射到不连续的字形 10, (hole), 25：

            ┌ idRangeOffset[] ┐ ┌───────── glyphIdArray[] ─────────┐
 slot:          0         1         2         3         4
            ┌─────────┬─────────┬─────────┬─────────┬─────────┐
 value:     │    4    │    0    │   10    │    0    │   25    │
            │ iRO[0]  │ iRO[1]  │ glyph A │ (hole)  │ glyph C │
            └────┬────┴─────────┴─────────┴─────────┴─────────┘
                 └─ "+4 bytes = +2 slots from MY slot" ─▶ slot 2 (start of this run)
                       then step +(c − startCode) slots into it
  • 'A' → 槽位 2 + 0 = 10
  • 'B' → 槽位 2 + 1 = 0 ⇒ .notdef（那个洞）
  • 'C' → 槽位 2 + 2 = 25

写成对这条合并带子的单个索引（idRangeOffset[idx] 就住在槽位 idx）：

slot = idx + idRangeOffset[idx]//2 + (c − startCode[idx])
       └─┬─┘   └────────┬────────┘   └────────┬────────┘
   self-relative base   bytes → slots      step into the run
gid  = strip[slot]
gid  = 0  ?  .notdef   :   (gid + idDelta[idx]) & 0xFFFF
要避开的那个两字符 bug 基址是 idx（偏移是自相对的——从那个单元量起）， //2 负责把字节换算成槽位。写成 val + val//2（保留 字节值），或者丢掉 idx 基址，都会让你越过目标几千个单元—— 而且它只咬 idRangeOffset != 0 这条路径，简单路径照常“能用”，把问题藏得严严实实。
这款字体实际怎么用 在它的 1532 个 format 4 分段里，564 个走 glyphIdArray 路径——而且这 564 个全部 idDelta = 0，内部没有洞。所以在这里 glyphIdArray 纯粹用于散乱的字形 ID（比如 ASCII，字形没按码位顺序排），从不用于 洞，而这条路径上的 idDelta 什么也不干。0 ⇒ .notdef 规则依然是 强制的——正是它允许洞的存在——这款字体只是恰好一个洞也没用。
2.8编程题

不走 .cmap 捷径，把一个码位手工穿过 format 4。拿到 一张 format 4 子表的原始字节，自己用 struct 解析表头和 四个数组，然后选一个字符（试试 永 U+6C38，想走简单路径就 选个数字），按 idRangeOffset==0 / !=0 规则一路解析到字形 ID。用 getBestCmap()（或 getGlyphID）验证。

卡住了？给个方向（不给完整答案） 子表能自我序列化：raw = subtable.compile(font) 给出的正是那份字节。 表头是七个大端序 uint16（">7H"）；四个数组按顺序跟在后面， 每个 segCount 条——但 idDelta 是带符号的 （">…h"），还要记得 endCode 后面那 2 字节的 reservedPad。 指针路径的偏移是从 idRangeOffset[i] 自身的地址量起的。
✓ 核对答案 永（U+6C38）的正确结果是 gid 9078——用 font.getGlyphID('永') == 9078 确认。（像 '5' U+0035 这样的数字走 idRangeOffset==0 的简单路径：gid = 0x35 + 210 = 263。）真正的检验标准是： 无论你选了哪个字符，手算出的 gid 都和 getGlyphID 一致。

Format 12：清醒的重新设计

format 12 覆盖完整的 32 位空间，而且简单得多。它的表头 16 字节，之后是一列 平铺的 12 字节分组：

字段类型值（这款字体）
formatuint1612
reserveduint160
lengthuint32166924
languageuint320
nGroupsuint3213909
groups[ ]3×uint32(startCharCode, endCharCode, startGlyphID)

每个分组的意思是“码位 startChar … endChar 映射到从 startGlyphID 开始的连续字形 ID”。查表不值一提：找到满足 startChar ≤ c ≤ endChar 的分组，然后 gid = startGlyphID + (c − startChar)。没有 idDelta，没有 idRangeOffset，没有指针把戏。

group for 一: startChar=U+4DBE  endChar=U+4E49  startGID=3105
gid = 3105 + (0x4E00 − 0x4DBE) = 3105 + 66 = 3171   → "u4E00"
证明你解析对了的交叉验证 format 4 经带符号的 idDelta 算出 一 → gid 3171；format 12 经分组算术算出 一 → gid 3171。两种完全不同的 编码，同一个答案——因为它们编码的是同一份映射。当你的手工解析和 另一张子表一致（也和 getGlyphID 一致）时，你就知道字节读对了。
