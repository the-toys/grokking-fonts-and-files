Module 02

cmap & glyph IDs

A font does not know about letters. Internally it knows only glyph IDs — small integers 0 … numGlyphs−1. The cmap table is the bridge from the Unicode codepoints in your text to those numbers. It is the first step of turning text into pixels, and deliberately not the last.

concept

How a character becomes a glyph

Everything a font can draw is identified by a glyph ID (often "GID"): an integer from 0 to numGlyphs−1. In this font numGlyphs is 46830, so the valid GIDs are 0 … 46829. Two facts are fixed for every sfnt font:

  • Glyph 0 is always .notdef — the "missing glyph", usually drawn as an empty or crossed box (the tofu). It exists so that "no glyph" still has something to draw.
  • Text on the wire is Unicode codepoints (U+0041, U+4E00, …), not glyph IDs. Something has to translate. That something is cmap.

So the very first thing a renderer does with "A一" is look each codepoint up in the cmap to get a glyph ID, then carry those IDs forward to every later stage (outlines, metrics, positioning). The mental model:

"A"  → U+0041 ──cmap──▶ GID 36   ──glyf──▶ outline ──hmtx──▶ width
"一" → U+4E00 ──cmap──▶ GID n    ──glyf──▶ outline ──hmtx──▶ width
The transferable insight cmap is a pure lookup: codepoint → glyph ID, one input, one output, no context. It cannot see neighbouring characters. So anything that depends on context — turning f+i into an fi ligature, picking a positional form, swapping in a stylistic alternate — happens later, in the shaping engine via GSUB/GPOS (Module 6). Keep the boundary sharp: cmap answers "which glyph is this character", shaping answers "given the surrounding run, which glyphs should we actually use, and where".

Why a table of subtables?

The cmap table is not a single mapping. It is a list of subtables, each declaring an audience and an encoding. The reason is historical: different platforms wanted different character encodings, and OpenType chose to ship several mappings in one table rather than pick a winner. Each subtable declares:

  • a (platformID, encodingID) pair — who this subtable is for;
  • a format — how the codepoint→GID data is encoded.

A renderer scans the subtables and picks the best one it understands. Two platforms can share the exact same mapping data presented under two different (platformID, encodingID) labels — which is exactly what we'll see below.

platformID and encodingID

platformIDplatformcommon (plat, enc) pairs
0Unicode(0,4) / (0,6) = full Unicode
1Macintosh (legacy)(1,0) = Mac Roman
3Windows(3,1) = BMP, (3,10) = full Unicode

The pairs you meet constantly in modern fonts are (3,1) "Windows BMP", (3,10) "Windows full Unicode", and the Unicode-platform equivalents (0,4)/(0,6).

Subtable formats

formatcoverswhat it is
0256 codepointstiny legacy byte table (one byte per char)
4BMP, U+0000…U+FFFFsegmented 16-bit mapping
6a contiguous 16-bit range"trimmed array", a single dense run
12full 32-bit, all 17 planessegmented 32-bit mapping
14variation selectorsUnicode Variation Sequences

The two that matter most today are format 4 (the workhorse for the Basic Multilingual Plane — the first 65,536 codepoints) and format 12 (which extends the same segmented idea to the full 32-bit space, so it can reach the supplementary planes above U+FFFF). Format 0 and format 6 are narrow legacy encodings you'll see in older or Mac-targeted fonts; format 14 doesn't map characters at all — it handles "use this variant of that character" sequences.

walkthrough

The real cmap of WenKai

Here is the genuine cmap of LXGWWenKaiMonoGB-Regular.ttf: five subtables. nMappings is how many codepoints each subtable maps.

platformIDplatEncIDformatnMappingswhat it is
03443498Unicode platform, BMP only (16-bit)
041246510Unicode platform, full range (32-bit)
05140Unicode Variation Sequences (empty here)
31443498Windows, BMP (16-bit)
3101246510Windows, full Unicode (32-bit)

Read row by row:

  • (0,3) format 4 and (3,1) format 4 both map 43498 codepoints — every character this font has that lives in the BMP (U+0000…U+FFFF). One is labelled for the Unicode platform, one for Windows; the data is the same.
  • (0,4) format 12 and (3,10) format 12 both map 46510 codepoints — the BMP characters plus everything beyond U+FFFF.
  • (0,5) format 14 maps 0: the variation-sequence subtable is present but carries no sequences. It's a structural placeholder.
The transferable insight The gap 46510 − 43498 = 3012 is the count of characters this font supports that sit above the 16-bit BMP — supplementary-plane codepoints like CJK Extension B (which starts at U+20000). A format-4 subtable is physically incapable of addressing those: its codepoints are 16-bit. That is the entire reason a font ships both a format-4 and a format-12 subtable. (And note numGlyphs = 46830 > 46510 mapped codepoints — some glyphs, like .notdef and shaping-only alternates, have no codepoint pointing at them.)

Why duplicate the same mapping across two platforms?

The (0,·) rows and the (3,·) rows carry the same content. This is redundancy on purpose: a macOS text stack (CoreText) historically looked for a Unicode-platform subtable, while Windows (DirectWrite/GDI) looked for a (3,1)/(3,10) Windows subtable. By shipping both labels, the font maps correctly on either OS without the renderer having to guess. The cost is a little extra space; the benefit is portability.

Names are a convenience, IDs are the reality A renderer never works with the string "u4E00". The cmap returns a glyph ID (an integer); the human-readable name "u4E00" lives in the post table purely so tools and people can talk about glyphs. fontTools' getBestCmap() hides one more step: it auto-selects the best available Unicode subtable — here the format-12 one, because it covers the most codepoints — and hands you a plain {codepoint: glyphName} dict.
See it for yourself specimens/extract.py LXGWWenKaiMonoGB-Regular.ttf prints the subtable list and sample mappings. With fontTools: TTFont(path)["cmap"].tables gives the five raw subtables, each with .platformID, .platEncID, .format, and a .cmap dict.
.notdef and tofu Glyph ID 0 is reserved as .notdef in every sfnt font. When a renderer asks cmap for a codepoint the font does not contain, it gets back GID 0 (or no entry, which it treats as GID 0) and draws .notdef — typically the empty rectangle people call a tofu box. Seeing tofu means "this font has no glyph for that character", not "the text is broken".
binary deep-dive

Hand-parsing the bytes: format 4

Knowing the model isn't the same as having decoded it. Let's parse the real (3,1) format-4 subtable of WenKai straight from its bytes — once by hand, so the next time you see it you recognize it. The subtable starts with a 14-byte header; here are its genuine first bytes (big-endian, as always):

bytesfieldvaluemeaning

c3 c0length50112byte length of this whole subtable
00 00language00 = not language-specific
0b f8segCountX23064→ segCount = 1532 segments
08 00searchRange2048the same binary-search trio from Module 1 — here over the endCode array. segCount 1532 → 2¹⁰=1024 → ×2=2048; log₂1024=10; 2·1532−2048=1016.
00 0aentrySelector10

There they are again The searchRange / entrySelector / rangeShift fossil you dismantled in Module 1 is baked into every format-4 subtable — this time to binary-search the endCode array for the segment containing a codepoint. Same trick, same fields, different array. Recognizing it cost you nothing here because you'd already met it.

The four parallel arrays

After the 14-byte header, format 4 stores four arrays of segCount entries each (plus a glyph-index array at the end), laid out back to back:

header(14)
endCode[1532]        uint16  — last codepoint of each segment (sorted; last entry = 0xFFFF)
reservedPad(2)       = 0
startCode[1532]      uint16  — first codepoint of each segment
idDelta[1532]        int16   — SIGNED add-constant for the segment
idRangeOffset[1532]  uint16  — 0, or a byte-offset into glyphIdArray (the tricky bit)
glyphIdArray[...]    uint16  — shared pool the idRangeOffsets point into

Every format-4 subtable must end with a segment whose endCode is 0xFFFF — the sentinel that guarantees the "find first endCode ≥ c" search always terminates.

The lookup algorithm

  1. Binary-search endCode for the first segment i with endCode[i] ≥ c. (This is where the trio earns its keep.)
  2. If c < startCode[i], the codepoint falls in a gap → glyph 0 (.notdef).
  3. If idRangeOffset[i] == 0 (the easy path): gid = (c + idDelta[i]) & 0xFFFF. The whole segment is one contiguous glyph run.
  4. Else (the pointer path): the glyph id lives in glyphIdArray, and idRangeOffset[i] is a byte offset measured from its own slot into that array. Then add idDelta.

Two worked lookups (both verified)

Easy path — 一 (U+4E00). It lands in a segment with idDelta = −16797, idRangeOffset = 0:

gid = (0x4E00 + (−16797)) & 0xFFFF
    = (19968 − 16797) & 0xFFFF
    = 3171                       → glyph "u4E00"  ✓

(Digits work the same way: '0' U+0030 in segment 1 → (48 + 210) = 258.) The signed idDelta is just "this run of codepoints maps to a run of glyph ids shifted by a constant" — cheap and dense.

Pointer path — A (U+0041). It's in segment 2 (startCode=0x3a, idDelta=0, idRangeOffset=3092). The spec's formula computes a byte address inside the subtable:

addr = idRangeOffset[i]  +  2·(c − startCode[i])  +  &idRangeOffset[i]
     = 3092              +  2·(65 − 58)           +  9212
     = 12318
glyphIdArray @ 12318 = 1 ,  + idDelta(0)         = gid 1   → glyph "A"  ✓
(cross-checked: font.getGlyphID('A') == 1)
Why that bizarre "offset from its own slot" idRangeOffset is the most confusing thing in OpenType, and it's a fossil of C pointer arithmetic. In the original C, glyphIdArray sits in memory immediately after idRangeOffset, so *(&idRangeOffset[i] + idRangeOffset[i]/2 + (c − startCode[i])) reaches into it naturally. Encoding the offset relative to the array slot itself let 1990s code do one pointer add with no separate base address. We're paying for that cleverness with confusion ever since — which is exactly why the next format threw it out.

Format 12: the sane redesign

Format 12 covers the full 32-bit space and is dramatically simpler. Its header is 16 bytes, then a flat list of 12-byte groups:

fieldtypevalue (this font)
formatuint1612
reserveduint160
lengthuint32166924
languageuint320
nGroupsuint3213909
groups[ ]3×uint32(startCharCode, endCharCode, startGlyphID)

Each group says "codepoints startChar … endChar map to consecutive glyph ids beginning at startGlyphID." Lookup is trivial: find the group with startChar ≤ c ≤ endChar, then gid = startGlyphID + (c − startChar). No idDelta, no idRangeOffset, no pointer games.

group for 一: startChar=U+4DBE  endChar=U+4E49  startGID=3105
gid = 3105 + (0x4E00 − 0x4DBE) = 3105 + 66 = 3171   → "u4E00"
The cross-check that proves you parsed it right Format 4 gave 一 → gid 3171 via signed idDelta; format 12 gives 一 → gid 3171 via group arithmetic. Two completely different encodings, same answer — because they encode the same mapping. When your hand-parse agrees with the other subtable (and with getGlyphID), you know you read the bytes correctly.
tooling & introspection

Exploring the font yourself

You will not remember every method name — nobody does. The skill is interrogating the object instead of looking it up. Your three-line recipe for any fontTools object:

type(obj)                                   # what class is this?
[a for a in dir(obj) if not a.startswith("_")]   # what can it do?
help(obj.some_method)                       # what does that do?

Run that on font['cmap'] and you discover it's a table__c_m_a_p whose public surface — the answer to your "what else besides .tables?" — is:

on font['cmap']returns / does
.tableslist of the raw subtable objects
.getBestCmap()auto-pick best Unicode subtable → {codepoint: glyphName} dict
.getcmap(plat, enc)fetch one specific subtable by (platformID, platEncID)
.buildReversed()invert it → {glyphName: {codepoints}} (e.g. {'space': {32}})
.compile(font)serialize back to bytes — the very bytes we hand-parsed above
.toXML() / decompile()the TTX text form / parse from bytes

And one subtable (a cmap_format_4) exposes: .cmap (the {codepoint: glyphName} dict), .format, .platformID, .platEncID, .language, .length, .isUnicode(), .isSymbol(), .getEncoding(), .compile(), .toXML(). So .compile(font) is your door from the tidy Python object down to the raw bytes — that's exactly how the deep-dive above got its data.

2.5concept

Where cmap stops and shaping starts. Consider the single precomposed character ﬁ (U+FB01, the "fi" ligature) versus typing the two characters f + i. Which one is resolved by cmap, and which one needs the shaping engine? Explain the boundary.

Show solution U+FB01 is a real codepoint, so cmap maps it directly to the single fi ligature glyph — one lookup, done. Typing f then i gives two codepoints (U+0066, U+0069); cmap maps each to its own glyph independently, because cmap has no notion of "these two are adjacent". Fusing them into one ligature glyph is a contextual decision, which is the job of GSUB (the liga feature) in the shaping stage. That is the boundary: cmap = context-free character→glyph; shaping = context-dependent glyph→glyph(s). We cover this in Module 6.
2.6concept

The special glyph. Why is glyph ID 0 special, and what appears on screen when a codepoint has no cmap entry?

Show solution Glyph ID 0 is reserved in every sfnt font as .notdef — the designated "missing glyph". When you request a codepoint the font does not map, the lookup yields GID 0 (or no entry, treated as GID 0), so the renderer draws .notdef: usually an empty or crossed rectangle, the tofu box. It signals "this font has no glyph for that character" — the text and the encoding are fine; coverage is what's missing.
