模块 06

整形与布局

到这里，你的字体已经能把字符映射到字形（模块 2）、画出轮廓（模块 3）、知道每个字形有多宽（模块 4）。但真正的排版远不止“一个字符一个字形，逐个排开”。整形（shaping）就是在落笔绘制之前改写并重新摆放字形串的那一步——把 f+i 合成一个 fi，把 AV 往一起收一收，或者把某个字形换成竖排用的形式， 好让 CJK 文字从上往下排。

练习是穿插式的——每道题就放在它考察的概念下面。 概念题带答案；编程题带可折叠的核对说明。

概念

字形如何组合与移动

回顾一下目前的流水线。文本以 Unicode 码位的形式进来。 cmap 表（模块 2）把每个码位转成字形 ID，得到 最初那串朴素的字形 ID。随后按 hmtx（模块 4）给出的前进宽度 画出轮廓（模块 3）。中间缺的那一步就是整形：

text (Unicode)
   │  cmap            ← codepoint → glyph-ID  (Module 2)
   ▼
glyph-ID run (naïve)
   │  SHAPING         ← GSUB rewrites + GPOS repositions   ◀── this module
   ▼
final positioned glyph run
   │  outline fill at advances  (Modules 3 + 4)
   ▼
pixels

整形介于 cmap 与最终摆放之间。驱动它的是两张表—— 一张管替换，一张管定位——两者几乎总是结伴出现：

  • GSUB——Glyph SUBstitution（字形替换）。把字形 ID 换成别的 字形 ID。连字（f+i → fi）、上下文 替代、组合/分解、分数拼合（1/2 → ½）、 上标下标、风格替代、竖排形式切换，全住在这里。字形的 数量可以改变。
  • GPOS——Glyph POSitioning（字形定位）。只移动字形，不改变 它们是哪些字形。字距调整、附标挂基字与附标叠附标（各种附加符号）、 连笔衔接、竖排度量微调都发生在这里。字形 ID 原封不动； 变的只有 x/y 偏移和前进宽度。
6.1概念题

给流水线排序。把下面四个阶段按执行顺序排好，并各用一句话 说明它做什么：GPOS、cmap、GSUB、轮廓填充。

看答案 1. cmap——把每个 Unicode 码位转成基础字形 ID。 2. GSUB——替换字形 ID（连字、替代字形、竖排 形式）：字形串长度可能改变。3. GPOS——移动字形 （字距调整、挂附标、微调度量），不改变它们是哪些字形。 4. 轮廓填充——在每个字形最终的位置/前进宽度上画出它的轮廓。 GSUB 先于 GPOS：先决定用哪些字形，再决定摆在哪里。

两张表内部共用同一套层级结构：

Script  (e.g. latn, hani)
  └─ LangSys  (language system, often just the default)
       └─ Features  (4-letter tags: 'liga', 'kern', 'calt', 'vert' …)
            └─ Lookups  (the actual substitution / positioning rules, run in order)

特性（feature）是应用程序可以选择开启的四字母 tag；tag 本身只是个名字，指向一个或多个 lookup——真正的规则存在 lookup 里， 按顺序执行。有的特性默认开启（除非明确要求关闭，引擎都会应用——如 ccmp、calt）；有的则按需开启，只在排版 请求时才生效（如 frac、sups）。

规则到底谁来执行 字体只负责存放 GSUB/GPOS；真正执行它们的是整形引擎。 开源世界事实上的标准是 HarfBuzz——Chrome、Firefox、Android、 LibreOffice 背后都是它。操作系统自带的对应物是 CoreText（macOS / iOS）和 DirectWrite（Windows）。它们读取文种和特性，逐个走 lookup， 输出最终摆放好的字形串。

拉丁文字跳过整形，顶多难看，还能读。复杂文种就没得 商量了：阿拉伯字母必须连写，并按位置取词首/词中/词尾/独立四种形式；许多 印度系文字还要重排元音、拼合辅音连体。在这些文字里，cmap 的原始输出——一个码位一个字形、从左到右——是真正读不了的。整形决定的是文本 对不对，而不只是好不好看。

6.2概念题

为什么有些文种绕不开整形。纯英文靠 cmap 的原始输出大体还能凑合，阿拉伯文和印度系文字却真正离不开整形引擎。 为什么？

看答案 拉丁文字的码位本来就按视觉顺序排列，每个字母基本只有一种形态，所以 一码位一字形是可读的（损失的只有连字和字距调整——纯属观感）。阿拉伯字母要 连写，形态随位置变化（词首/词中/词尾/独立），引擎必须选对 上下文形式并把字母接起来。许多印度系文字还会重排——排在 辅音后面输入的元音，可能要显示在它前面——并把辅音簇合并成连体 字形。这些都是整形引擎经由 GSUB/GPOS 完成的强制性替换与重排； 没有它，cmap 的原始输出顺序不对、形态也不对——不是不够精致， 是读不了。
两张容易搞混的“映射” cmap 是码位 → 字形：一次查表、不看上下文，是整个流程的入口。 GSUB 是字形 → 上下文中的一个或多个字形：属于排版层，在其后应用。 还要提防第二处撞名：GSUB 里 ccmp 的“组合”是一种整形 替换（把这几个字形 ID 换成那一个），跟模块 3 的 复合字形（一条轮廓把另一条轮廓当组件复用）完全是两码事。 词一样，层不一样。
6.3概念题

cmap 与 GSUB。cmap 的映射和 GSUB 的替换有什么区别？各举一个具体例子。

看答案 cmap 是一张单一的、不看上下文的表，把一个码位映射到一个基础 字形——例如 U+4E00“一”→ 字形 u4E00。GSUB 接过这串 字形，结合上下文改写它，连数量都可能变——例如 f+i 的两个字形 → 一个 fi 连字，或者开着 frac 时一串数字坍缩成 ½。cmap 回答“这个字符一开始 是哪个字形？”；GSUB 回答“看看周围的字形和请求的特性，它 该变成什么？”。
6.4概念题

两种“复合”。GSUB 的 ccmp 叫字形 组合/分解，模块 3 又有复合字形。两者都在“拼” 东西，到底差在哪？

看答案 层次不同。GSUB ccmp 是一种整形替换：它 改写的是字形 ID 串——比如把一个预组合字符拆成基字 + 附标 两个字形（或反过来合并），好让后面的附标定位 lookup 干净利落地把它们挂上去。 它操作的是字形序列，发生在绘制之前。模块 3 的复合字形 是 glyf 内部的轮廓复用机制：一个字形的轮廓 通过引用另一个字形作为组件、放到某个偏移处来定义（例如 dieresis 在 x=500 处复用 dieresiscomb）。那说的是单个字形的形状 怎么存，跟改写字形串无关。ccmp 改变用哪些字形；glyf 复合改变 一个字形怎么画。
解读

霞鹜文楷等宽 GB（LXGW WenKai Mono GB）的 GSUB / GPOS 拆解

直接从两张布局表里读出来，LXGWWenKaiMonoGB-Regular.ttf 实际携带的 内容如下：

GPOS DFLT bopo grek hani kana latn 3 halt vert vhal

回想模块 1 的表目录：GPOS 只有 420 字节，GSUB 也不过 5564 字节——跟 24.7 MB 的 glyf 块比起来微不足道，却 装下了全部排版逻辑。

为什么恰好是这套特性组合 这款字体把 GSUB vert 和 GPOS vert、vhal、 halt 打包在一起，原因只有一个：CJK 文本可以竖排。 排版引擎切到竖排模式时，GSUB vert 把字形换成竖排 形式（比如旋转后的括号和标点），GPOS vert / vhal / halt 则为新的方向调整度量。定位这一侧 依赖模块 4 讲过的 vhea 和 vmtx 表—— 竖排头部和竖排前进宽度。替换与定位协同作战，正是 GSUB + GPOS 的分工，而这里它们服务的正是竖排 CJK。

6.5编程题

用 fontTools 遍历布局表。用 python 打开文楷，对 GSUB 和 GPOS 各打印：文种 tag、特性 tag、lookup 数量。你的输出应能 复现解读部分的表格（各 6 个文种；GSUB 12 个 lookup / 8 个特性；GPOS 3 个 lookup / 3 个特性）。

对 GSUB 和 GPOS 各自：拿到表对象，收集文种 tag 和特性 tag，数出 lookup 数量，然后把三者都打印出来，跟上面解读部分的 数字对上。

卡住了？给个方向（不给完整答案） 层级结构可以从 f['GSUB'].table.ScriptList / .FeatureList / .LookupList 一路访问。每个 ScriptRecord 有 .ScriptTag；每个 FeatureRecord 有 .FeatureTag；lookup 数量 看 LookupList.LookupCount（或 len(LookupList.Lookup)）。

把你的代码和输出贴给 AI 导师，让它核对 tag 和数量 是否与样本一致。

解读 · 实际效果

GSUB 实战：替换前 → 替换后的字形串

特性 tag 很抽象，直到你亲眼看它改写一串字形。下表每一行都是这款 字体里真实发生的替换：把字符串丢给 HarfBuzz 整形，再和朴素的 cmap 输出做 diff 得来。每行读作字形 ID 原来是什么 → GSUB 把它们变成了什么。盯住数量的变化——那是 GSUB 的签名。

特性你输入字形串：替换前 → 替换后发生了什么
frac1/2[one, slash, two] → [onehalf.full]3 个字形坍缩成一个 ½
frac3/4[three, slash, four] → [threequarters.full]同一条规则，分子分母随意 → ¾
afrc1/2[one, slash, two] → [one_fraction_two.nut]竖式“nut”分数（分子叠在分母上）
sups2[two] → [twosuperior]1→1 换成升起的上标 ²
subs3[three] → [threeinferior]1→1 换成下沉的下标 ₃
ccmpA + ◌̀[A, gravecomb] → [Agrave]组合：基字 + 独立的组合附标合并成一个预组合的 À，音符落得严丝合缝
ccmp⸺（两 em 长破折号）[twoemdash] → [emdash.ccmp, emdash]分解：1 个字形展开成无缝衔接的多段破折号，随便多长都连成一线
vert。（竖排文本）[ideographicfullstop] → […verticalideographicfullstop]换成该字形的竖排形式（句号挪到字格的右上角）
salt（用户自选）[glyph] → [glyph.salt]由用户自选的风格替代造型——同一个字符，不同的画法

.full / .nut / .salt 这些后缀只是字形名—— 字体专门为对应特性准备的替代字形。GSUB 的全部工作，就是决定每个位置 最终落到哪个字形名上。

GSUB 替换类型——规则的四种形态 上面每条替换都属于寥寥几种 lookup 类型之一：单一（1→1，如 sups、vert、salt）、多重（1→多，即 ccmp 破折号的分解）、连字（多→1，frac 和 ccmp 那个 À 的组合），以及上下文/链式（邻居匹配才 替换，calt 和大多数文种逻辑靠的就是它）。文楷的 12 个 GSUB lookup 恰好就是这几类（单一 ×5、多重、连字 ×3、链式 ×2）。

跳出这一款字体，同一套机制还撑起那些经典特性——文楷虽然没 声明，但值得知道：

  • liga——f+i → 单个 fi 字形，f 的钩与去点的 i 画成一个整体（多→1，和 frac 同类）。
  • 阿拉伯文连写——一个字母按位置分出四种字形 （独立/词首/词中/词尾）；整形器选对上下文形式，字母才连得上。
  • 印度系文字重排——排在辅音之后输入的元音会被替换并 挪到它前面，辅音簇还会熔成连体字形。这是强制性的，不是装饰。

GPOS 实战：字形不变，位置在动

GPOS 从不改变你手里是哪些字形——串的长度固定不变。它改的是 x/y 偏移和前进宽度。文楷的 GPOS 全围着竖排 CJK 度量转； 下面是 HarfBuzz 给出的真实位置 diff（同一个字符的横排 vs 竖排）：

特性字形横排竖排发生了什么
vert / vhal。x 前进宽度 1000，无偏移y 前进宽度 1000（向下），y 偏移 −880小句号被抬到 em 字格的右上角，笔尖改为沿字列向下走，而不是向右

再看外面世界依赖的那些 GPOS 经典（同样不在文楷里，但用的是同一张表）：

  • kern——AV：两个字形都原样保留，只把 V 向左移约 80 单位，让两个形状咬合到一起。纯定位，零替换。
  • mark（mark-to-base，附标挂基字）——组合附标靠 锚点（对准字母中心）而非前进宽度来定位，这样 e+◌̈ 不管 e 多宽，分音符都正好落在正中。
  • mkmk（mark-to-mark，附标叠附标）——在第一个附标上方再叠 第二个（比如越南语），每一个都以前一个附标为基准排开。
  • curs（连笔衔接）——把一个字形的出笔锚点 焊到下一个字形的入笔锚点上，让阿拉伯文的连笔一气呵成。
一句话判别法：数量变了没？ 字形串变长或变短了，那是 GSUB（连字、分解、 分数）。字形还是那些、只是待的位置变了，那是 GPOS（字距调整、挂附标、竖排度量）。GSUB 决定用哪些字形，GPOS 决定摆在哪里——顺序永远如此。
自己复现 这几行都是真实的 hb.shape() 输出。装好 HarfBuzz （uv pip install uharfbuzz）后，整形一个字符串，再把字形名和 font.getBestCmap() 做 diff；也可以直接跑命令行 hb-shape --features=frac LXGWWenKaiMonoGB-Regular.ttf "1/2"，看着 [one, slash, two] 变成 onehalf.full。再加 --direction=ttb，就能看到 vert 替换生效。
6.6概念题

GSUB 还是 GPOS？对下面每个 tag——calt、frac、 vert——说出它属于哪张表、产生什么可见效果。

看答案 calt = GSUB——根据相邻字形做替换（上下文 替代造型）。frac = GSUB——把数字/斜杠/数字的字形串 换成单个分数字形，如 1/2 → ½。vert 在这款字体里 两边都有：GSUB 的 vert 把字形换成竖排形式，GPOS 的 vert 为竖排微调该字形的位置/度量。calt/frac 这两个改变的是用哪个字形；vert 既换字形， 还要挪它的位置。
