模块 03

轮廓

字形就是一组封闭路径，由渲染器填充。我们按最自然的顺序层层递进： 形状存在哪里（glyf）、怎么找到某一个字形的数据（loca）、 两张表如何配合、解出来的数字是什么意思， 最后徒手解码原始字节。

练习穿插在正文里。概念题带答案；编程题带一个可折叠的 ✓ 核对答案。

概念 · 1

glyf——形状存放的地方

glyf 表是一整块大数据，装着所有字形的轮廓，其中每个字形占一条变长记录， 前后紧挨着排。一条记录把字形描述成若干轮廓环——由点构成的封闭回路——外加一个包围盒。 渲染器填充这些回路围出的区域，而不是只描边。

这些点描出的是哪种曲线，由字体采用的技术路线决定，而技术路线由 模块 1 里的 sfnt 魔数宣告：

  • TrueType（本字体，魔数 00 01 00 00）：轮廓放在 glyf 里，用二次曲线。（这是什么意思，第 4 步拆开讲。）
  • CFF / CFF2（魔数 OTTO）：轮廓改放在 CFF  表里， 用三次曲线。压根没有 glyf。

眼下只需记住一句：glyf = 轮廓数据；每个字形一条紧密排列的 记录。下一个问题是，怎么在这 24.7 MB 的大块数据里找到某一个字形。

3.1概念题

轮廓放在哪？你手上这款字体的 sfnt 魔数是 00 01 00 00。它的轮廓在哪张 表里、用哪种曲线？换成魔数为 OTTO 的字体， 又会用哪张表、哪种曲线？

看答案 00 01 00 00 ⇒ TrueType ⇒ 轮廓在 glyf 里（经 loca 定位）， 二次曲线。OTTO ⇒ 轮廓在 CFF  表里， 三次曲线，且没有 glyf/loca。
概念 · 2

loca——定位一个字形的字节

glyf 的记录是变长的（一个逗号只有几个点，一个汉字动辄几百个）， 中间又没有分隔符，没有地图就没法直接跳到第 i 个字形。 这张地图就是 loca——一个指向 glyf 内部的字节偏移数组， 每个字形一项，再多一项——共 numGlyphs + 1 项，其中 numGlyphs 是字体的字形总数（存在 模块 1 讲过的 maxp 表里；本字体是 46830，模块 2 用过这个数）：

loca:   off0      off1          off2       off3   …   offN+1   (N+1 entries, N = numGlyphs)
        │         │             │          │
        ▼         ▼             ▼          ▼
glyf:   ┌─────────┬─────────────┬──────────┬─ …              (variable-length records, packed)
        │ glyph 0 │   glyph 1   │ glyph 2  │
        └─────────┴─────────────┴──────────┴─ …

        each loca[i] marks the START of glyph i, so   glyph i's bytes = glyf[ loca[i] : loca[i+1] ]

有三条推论值得说破：

  • 为什么是 N+1 项：字形 i 的范围从 loca[i] 到 loca[i+1]，所以最后一个字形也需要一个位于它之后的偏移来标记结尾。
  • 空字形：如果 loca[i] == loca[i+1]，这个字形就是零字节—— 什么都不画（比如空格）。完全正常。
  • 两种存储格式：每项的宽度由一个字段决定，写作 head.indexToLocFormat（即“head 表的 indexToLocFormat 字段”）。head 是 OpenType 的字体头部表——一张经由 表目录找到的具名表，模块 1 列过，不是文件 offset 0 处那 12 字节 的文件头。0 = 16 位“短”偏移，存的是真实偏移 ÷ 2（省空间， 但最多只够到 128 KB 的 glyf）；1 = 32 位“长”偏移，直接存原值。 本字体用 1（它的 glyf 有 24.7 MB——短偏移远远 够不着）。
3.2概念题

数一数 loca，再读一个间隙。本字体 numGlyphs = 46830。(a) loca 一共有多少项？为什么多出一项？(b) 如果 loca[40] == loca[41]， 你能对字形 40 得出什么结论？

看答案 (a) 46831——字形 i 占 loca[i]…loca[i+1]，所以需要一个 越过最后一个字形（46829）的偏移来给它收尾。(b) 字形 40 是零字节——一个 什么都不画的空字形（比如空格）。
解读 · 3

glyf + loca 联手（以及我们的第一批真实坐标）

两张表合起来就能取出一个字形。拿 '一'（U+4E00）来试。先码位 → 字形 ID（走模块 2 的 cmap），再字形 ID → 字节范围（走 loca）：

name  = getBestCmap()[0x4E00]   = "u4E00"
gid   = getGlyphID(name)        = 3171
loca[3171]   = 931382      ┐  glyph 3171's bytes are
loca[3172]   = 931454      ┘  glyf[931382 : 931454]  →  72 bytes long

这 72 个字节就是轮廓本身。fontTools 会把它们解码成一个干净的对象，让你先不碰字节 也能读出形状：

from fontTools.ttLib import TTFont
font = TTFont(path)
g = font["glyf"]["u4E00"]
g.numberOfContours      # 1   (>0 simple glyph, -1 composite)
g.coordinates           # 21 ABSOLUTE (x,y) points: [(110,383), (837,420), …]
g.flags                 # parallel bytes; bit0 (& 1) = on-curve
g.endPtsOfContours      # [20]  last point index of each contour

所以字形 '一' 是 1 条轮廓环、21 个点，头几个坐标是 (110,383) (837,420) (861,422) (881,430) …。这些裸数字什么意思，第 4 步来解。 （而且没错——这跟表目录在整个文件层面做的导航一模一样，只是下沉了 一层：一个偏移索引，指向一块紧密排列的数据。）

概念 · 4

每个坐标是什么意思

现在到收获的时候了。我们手里有一串 (110, 383) 这样的点，每个点带一个 flag。要解码 两件事：点在哪里，以及它扮演什么角色。

哪里：字体单位，从基线量起

坐标以字体单位计：x 向右，从原点（左 缘）起；y 向上，从基线起——基线是文字坐在其上的那条线，位于 y = 0。unitsPerEm = 1000 是刻度（1 em = 1000 单位），所以 (110, 383) 就是向右 110 单位、基线上方 383 单位。

坐标带符号——也不被框死在 [0, 1000] 里 基线不是字形的底边，而是书写线。基线以下 的部分 y 为负——拉丁字母的降部（'g' → −175， 还有 'p'、'y'、'j'），以及向下探的汉字笔画 （'十' 的竖笔伸到 y = −86）。墨迹也可以落在原点左侧 （x 为负），或伸出 em 之外。1000 是一把尺子，不是包围盒——字形 并不被关在里面。（所以 g.coordinates 里出现 (537, −59) 时，那不过是 “x 537，基线下方 59 单位”。）

单位怎么缩放成像素，是模块 4 的全部内容；眼下只需把每个点 读作这把以基线为参照的尺子上的一个位置。

角色：曲线上点与曲线外点——顺带说清贝塞尔曲线是什么

每个点 flag 的第 0 位标记它是曲线上点还是曲线外点。要明白这 为什么重要，先看藏在底下的那一个想法：从 A 到 B 画一条直线，然后在旁边放下第三个点 C，把它当 磁铁——让线段朝 C 鼓起来，但仍从 A 出发、到 B 结束、永远 碰不到 C。这条被掰弯的线，就是二次贝塞尔曲线。

          C ●   ← off-curve "control" (the magnet: curve leans toward it, never touches)
         ·     ·
       ·         ·          ← the actual curve
  A ●·             ·● B
  on-curve          on-curve   (the curve DOES pass through A and B)
  • 曲线上点（A、B）——路径穿过它们；也叫 锚点（轮廓被钉在这些点上）。
  • 曲线外点（C）——路径只是朝它偏；也叫 控制点。C 离得远 → 鼓得大；C 落在 A–B 连线上 → 就是直线。

二次与三次之别 = 每段几块磁铁：二次曲线一个 控制点（TrueType）；三次曲线两个，两端各一（CFF），一段之内就够画出 S 形。 二次曲线一段画不出 S，所以 TrueType 要串好几段——同一个形状，TrueType 字形的点数比 CFF 多，原因就在这。

读一串 flag

一条轮廓环就是这些点连成的回路。曲线上/外的 flag 告诉你各段是什么：

  • 1 1——两个锚点相邻，中间没有磁铁 ⇒ 一条直边。
  • 1 0 1——锚点、控制点、锚点 ⇒ 一段曲线。
  • … 0 0 …——两个曲线外点相邻 ⇒ TrueType 在它们的中点隐含一个 曲线上点，于是一串控制点连成一条平滑的弧链。

于是 '一' 的前六个 flag 1 1 0 0 0 0 读作：锚点、锚点（顶部那条直边）， 然后连续四个控制点（右端的圆头），控制点之间夹着隐含的中点。下面把它 原原本本画在 em 方格上：

轮廓环、洞与环绕方向

一个字形是一条或多条封闭回路（轮廓环），渲染器填充它们的内部。 唯一的麻烦在回路嵌套时——'O' 的碗、 '口' 的框——渲染器得判断被围住的区域是墨迹还是洞。一条 规则包办所有情形。

规则本身

从待测的点向无穷远射一条射线，数它穿过多少条轮廓边—— 但要带符号：按边穿过射线的方向，每次相交记 +1 或 −1（一个方向 +1，反方向 −1——哪边算哪边只是 固定约定）。带符号的总和不为零，点就在内部；为零则在 外部。这就是非零环绕规则。（它有个更简单的表亲——奇偶规则： 丢掉符号，交点数为奇则填充；TrueType 用非零规则。）

为什么嵌套会产生洞

全部窍门就一个事实：内环画的方向与外环 相反。从每类点各射一条射线出去看看：

  • 在实心环带里（外环与内环之间）：射线只穿过外环的边—— 一次相交，总和不可能为零 ⇒ 填充。（它的符号听约定的 ——这里是 −1，因为外环顺时针——但要紧的只有 零与非零，符号本身从来无关。）
  • 在被围住的区域里：射线穿过内环的边又穿过外环的边，而 两环走向相反，两次相交符号相反、彼此 抵消（+1 加 −1 → 0）⇒ 空白——一个洞。
  • 在一切之外：没有相交 ⇒ 0 ⇒ 空白。
挖洞靠的是反向环绕，不是嵌套本身 把内环改成与外环同向来画，两次相交就取相同 符号——它们相加而非抵消（得 ±2，永远不为零）——那个“洞” 就被填实了。开洞靠的是方向；一条回路仅仅呆在另一条里面，还不够。

所以多条轮廓环并不意味着有洞：'永' 是三笔，走向全部 相同——三块彼此独立的实心墨迹，零个洞。而拉丁 'O' 恰恰就是 '口' 那幅图：一条外环，加一条反向的内环。真实例子如下：

为什么射线朝哪射都无所谓

这套做法理应让你起疑：要是我把射线朝上射而不是朝 右射，答案还一样吗？一样——永远一样，而看懂为什么正是 整条规则的心脏，所以我们仔细说。

先把射线忘掉。站在待测点 P，看一支笔把整条轮廓环 描一圈。只盯住从 P 指向笔尖的方向——一个 随笔移动而摆动的角度。笔回到起点时，这个角度也回到原处，但一路上它 可能已绕着 P 转了整数个整圈。这个整数——逆时针圈数 减顺时针圈数——就是 P 的环绕数。把这个 定义再读一遍：里面根本没提射线。它只取决于 P 和曲线本身。

射线法不过是数这些圈数的一种手段。轮廓每穿过你的射线一次， 笔的方向就正扫过射线所在的那一个固定角度——要么沿 +圈方向（记 +1）， 要么沿 −圈方向（记 −1）。整圈描完，带符号的相交数加起来恰好是净 圈数——环绕数。既然这个数的定义里没有射线，任何方向的任何 射线都必须报出同一个总数。射线只是数圈用的刻度线；刻度放在 哪，改变不了笔实际转了几圈。

同一个事实，换成抵消的看法 不用角度论证也能看到它。把射线从一个方向慢慢转向另一个方向。 带符号的相交数只可能在射线相切擦过曲线的那一瞬间变化 ——而在那里，两个交点总是成对出现（或成对消失），且符号相反： 射线探进边界又退出来，一个 +1 一个 −1，正好抵消。于是 射线转动时带符号的总和纹丝不动 ⇒ 所有方向一致。（奇偶规则也因同一个理由 成立：交点仍然成对出现和消失，所以计数的奇偶性同样 不会变。）

老实交代一个前提：射线要选一般位置——不能正好 穿过某个列出的点，也不能沿着某条边走。这类退化方向只有有限多个；真实 的栅格化器用固定的裁决规则绕开它们（比如“一条边只在它的下端点计一次”），保证 边界像素既不被数两次也不被漏掉。

渲染器实际怎么填充

逐点射线是“内部”的定义，不是算法——没有哪个栅格化器 给每个像素单独射线。它按行工作（扫描线转换）：

  1. 在一行像素上取一条水平扫描线（过该行中心）。
  2. 找出轮廓边与它相交的位置——只要 x 坐标，每个交点按方向标上 +1/−1。
  3. 把这些 x 排序，从左到右扫过去，维护一个环绕数累计和；它只在 交点处变化，累计和非零的区段全部填充。

一条扫描线一次答完一整行——“射线”就是这条扫描线，被该行所有像素 复用，而不是每个点重射一次。

那些别扭的情形（扫描线擦过顶点，或恰好压在水平边上）不是靠转动射线 解决的——而是靠一条固定的裁决规则：把每条边在 y 方向当作半开 区间（在下端点计数，上端点不计），并忽略严格水平的边。这样 每个共享顶点恰好被数一次，相邻区段干净地共享边界——不重画、 没有一像素宽的裂缝。（这跟 GPU 画三角形用的“top-left 规则”是同一回事，而它站得住脚正因为 任何射线方向都给出一致答案——我们刚证过。）

在此之上还叠着两层。抗锯齿：好的栅格化器不做非黑即白的内外判断，而是 计算每个像素的分数覆盖率（轮廓盖住了它多大一块）——靠子采样，或 按像素格累计精确的带符号面积（FreeType 的“smooth”栅格化器）——环绕逻辑不变， 只是从是/否改成量面积。Hinting（渲染微调）：小字号下， prep/fpgm/gasp 这套机制会在填充之前把轮廓对齐到像素 网格上，让笔画保持锐利。

细则：方向、角色与定向

方向不等于角色。一条轮廓环的方向（顺时针/逆时针， 按 y 向上判断）恰好是其带符号（鞋带公式）面积的正负——可以逐环 检验。但它到底是外边界、洞，还是洞里的实心岛，是一个 包含关系问题，符号给不了答案。本字体遵循的约定 （TrueType；PostScript/CFF 正相反）：最外层轮廓环顺时针，每嵌套一层， 环绕方向翻转一次（CW、CCW、CW……），实心与空心由外向内交替。“外环 CW、洞 CCW”只是两层的特例——'回' 嵌套四层，它的第三条环虽是顺时针， 却是内环，不是最外层。

“顺时针”什么时候才有定义？沿一条永不自交的回路走一圈，你面朝的 方向恰好转过一整圈——+360°（CCW）或 −360°（CW）； 这个正负号就是定向，带符号面积能把它算出来。路径一旦 自交（8 字形的两个环各转一个方向；对称的 8 字带符号面积 干脆等于 0），或者退化成零面积，定向就没了定义。干净的字形里两种情况 都不会出现——况且栅格化器根本不需要全局标签，因为环绕数 是由局部相交逐点定义的。

3.5概念题

为什么随便哪条射线都行。(a) 用一句话说明：为什么朝任意 方向射线，对点 P 的内外判断都一样？(b) 射线必须满足的唯一条件是什么？不满足会 出什么岔子？

看答案 (a) 两条理由任选。干净的一条：带符号的相交数就是 P 的 环绕数——描完一圈轮廓时，从 P 指向动点的方向净转过的 整圈数——这个量的定义里没有射线，自然不依赖你朝哪 射。动手的一条：缓缓转动射线，交点只会以互相抵消的 +1/−1 对出现或消失，总和从不改变。(b) 射线必须处于一般位置：不能 正好穿过轮廓点，也不能沿边而行——否则一次相交就模棱两可 （数成零次、一次还是两次都说得通），栅格化器用固定的裁决规则避开这一点。

徒手解析一个字形的字节

库我们用过了；现在自己解码原始字节。'一' 有 21 个点（手算太 多），所以换用最小却五脏俱全的真实字形：'─'（U+2500）， 一个只有 4 个点的矩形。先导航到它的字节（表偏移来自模块 1： glyf @ 592352，loca @ 405028）：

name         = getBestCmap()[0x2500]   = "boxdrawingslighthorizontal"   (not "u2500"!)
gid          = getGlyphID(name)        = 869
loca[869]    = 153477                   ┐ length = 153501 − 153477 = 24 bytes
loca[870]    = 153501                   ┘ file offset = 592352 + 153477 = 745829

注意字形名并不固定——要经 cmap 解析（getBestCmap()[0x2500]）， 别想当然以为是 "u2500"。'一' 恰好叫 "u4E00"，但这个模式没有保证。

怎么真正靠 loca 抓到字节 loca 是一个含 N+1 个偏移的数组（末尾多一个哨兵）。字形 gid 的数据是半开区间 [loca[gid], loca[gid+1])——要读该字形 自己的偏移和下一个偏移，两者之差就是长度。末尾那个 哨兵正是数组比字形数多一项的原因：最后一个字形也需要一个 gid+1 来收尾。有三件事必须做对：

(1) loca 的偏移相对于 glyf 表，不相对于文件。在原始文件里 seek 时要加上 glyf 表自身的偏移（来自模块 1 的表目录）： file_off = glyf_offset + loca[gid]。
(2) loca 的短格式与长格式（head.indexToLocFormat）：0 = 短格式，每个偏移是一个 uint16，存的是偏移 ÷ 2（用时乘 2，字形按 2 字节对齐）；1 = 长格式，每个是 uint32 字节偏移，直接用。本字体 是长格式。fontTools 的 font['loca'].locations 两种格式都直接给你解码好的 字节偏移——那个 ÷2 只在你亲手解析 loca 字节时才会咬人。
(3) 长度为零 ⇒ 没有轮廓。如果 loca[gid] == loca[gid+1]，这个字形 完全没有 glyf 数据——比如 space 这样的空白字形。什么都别读；别去解析头部。 
import struct
glyf_off   = font.reader.tables['glyf'].offset    # 592352, from the sfnt directory
loca       = font['loca'].locations           # decoded offsets, len = numGlyphs + 1
gid        = 869
start, end = loca[gid], loca[gid+1]            # 153477, 153501
length     = end - start                     # 24   (0 ⇒ blank glyph, stop here)
with open(path, "rb") as fh:
    fh.seek(glyf_off + start)                # 745829
    raw = fh.read(length)
# raw = 00 01 00 00 01 68 03 e8 01 90 00 03 00 00 11 35 21 15 03 e8 01 68 28 28

'─' 是最省事的情形——一条轮廓环，没有 hinting。通用的简单字形布局里，上面那两行 其实各藏着一个变长字段：

两个不定宽的字段 (1) endPtsOfContours 是 n 个 uint16，共 2·n 字节——'─' 只有一条轮廓环 （2 字节），而像本字体 .notdef 那样的 5 环字形带的是 00 21 00 46 00 93 00 c9 00 d1 = (33, 70, 147, 201, 209)，十个字节。每个值 是该环最后一个点的索引（累计值，不是数量）：环 0 是点 0–33， 环 1 是点 34–70，依此类推。关键在于，最后一项 + 1 就是总点数 （209 + 1 = 210）——读 flags 之前必须先拿到这个数，因为每个点 一个 flag。

(2) instructionLength 后面跟着同样多字节的 hinting。如果 instructionLength = m 不为零，恰好 m 字节的 TrueType hinting 字节码就夹在 instructionLength 和 flags 之间。它们在特定渲染字号下把点往像素网格上推—— 并不改变存储的轮廓，所以徒手解析时直接跳过 m 字节去读 flags 就行。（本字体的每个字形都是 instructionLength = 0——大型中文字体常见如此：与其给几万个字形逐个手工 hint，不如交给渲染器的自动 hinter——这就是 '─' 显示 00 00、flags 紧随其后的原因。）

flags 与增量编码的坐标

头部之后是 flags（每点一个，可按行程压缩），然后是坐标。flag 字节是一个位域。x 增量由一对位（第 1、4 位）控制；y 增量由镜像 的一对（第 2、5 位）控制：

20x04Y_SHORT同上，作用于 y
30x08REPEAT下一个字节 = 再有多少个点重复这个 flag
40x10X_SAME_OR_SIGN含义取决于 X_SHORT——见下面的 2×2 表
50x20Y_SAME_OR_SIGN同上，作用于 y
每点一整个字节，不是 6 个位 每个 flag 都是完整的一字节。上面只定义了第 0–5 位（第 6–7 位保留）， 但解析器仍然每个点读满一字节——flags 按字节对齐，绝不 跨点打包位流，所以不存在“剥完 6 位再跨进下一个字节”的情况。（正是 这种字节对齐让 REPEAT 得以成立：它复制的是一整个相同的 flag 字节。）

一个字节之内，“第 N 位”指权值为 2N 的那一位，从 最低有效端数起：第 0 位 = 0x01 是 LSB，第 5 位 = 0x20。 用 flag & mask 测试某一位。从 LSB 往上剥 0x35 = 0011 0101： 第 0、2、4、5 位置位（即 ON_CURVE、Y_SHORT、X_SAME_OR_SIGN、Y_SAME_OR_SIGN—— 注意 0x35 & 0x02 = 0，X_SHORT 没有置位）。

第 4 位最容易让人栽跟头：它有两个互不相干的含义，用哪个由第 1 位（X_SHORT） 决定。规范里的名字把两半都写全了—— X_IS_SAME_OR_POSITIVE_X_SHORT_VECTOR：当增量是短格式时，第 4 位是它的 符号；当不是时，第 4 位表示“与前一点相同”（增量为零）。 于是这两个 x 位合起来，恰好从四种情形里选定一种：

第 1 位（X_SHORT）第 4 位读取字节数x 增量
111+（该字节，0–255）

（短格式是一个无符号字节，符号由第 4 位补上——1 个字节因此仍能覆盖 ±255。y 增量的规则一模一样，用第 2、5 位。）

我们的四个 flag 字节 11 35 21 15（没有 REPEAT 位，一点一个）：四个的第 0 位 都置位 ⇒ 全是曲线上点。后面的坐标字节怎么读，还有两条规则管着：

读坐标流的两条规则 (1) 增量（累加）：每个存储值都是相对前一 点的步长，所以点的绝对坐标是累计和——从 0 起 （第一步从原点量起），逐个把增量加上去；x 和 y 各自独立 累加。(2) 先全部 x，再全部 y：所有 x 增量在前，所有 y 增量在后 ——不交错。每个增量占多宽（0、1 或 2 字节）、符号是什么，直接 查上面那张 2×2 表。

现在解码末尾 6 个字节 03 e8 01 68 28 28。分两趟——先读完所有 x 增量，再读所有 y 增量——让一个游标在字节间走，每步恰好吃掉 flag 规定的宽度。第 1 步：从四个 flag 读出每个点 x、y 各占多宽 （查 2×2 表）：

原始字节与 g.coordinates——别把两层搞混 这套增量/累加编码只存在于原始 glyf 字节里。fontTools 的 g.coordinates（第 3 步）已经是绝对坐标——加载字形的 那一刻它就把累加做完了。所以 '十' 的 g.coordinates = [(536,386), (537,−59), …] 是可以直接拿去画的位置（画出来 也对）；要是把它们当增量再累加一遍，你会得到 (536,386), (1073,327), …——x 超出 1000 em，纯属胡闹。只有像本节这样 亲手解析原始字节时，才需要自己做累加。
两个坑 坐标是先全部 x 再全部 y（读完所有 x 增量才轮到 y），而第 4/5 位 看上下文——对应的 SHORT 位置位时是“符号”，没置位时是 “增量为零”。把它们当成一成不变的单一含义，坐标就会漂。

字节里的复合字形：分音符

复合字形自己不画任何轮廓——它摆放别的字形，每个占一条 组件记录：一个 16 位的 flags 字、被引用字形的索引，然后是摆放 参数（可选地再跟一个变换）。带间距的分音符 numberOfContours = −1，随后 就是一条这样的记录：

这个 flags 字支配着它后面的一切。正如简单字形的坐标位决定每个 增量的宽度，这 16 个位让组件记录成为变长的——它们 决定参数有多宽、参数是什么意思、后面是否还跟着变换和更多 组件：

位掩码名称对记录的影响

我们这条是 0x0007 = 第 0+1+2 位 ⇒ ARG_1_AND_2_ARE_WORDS | ARGS_ARE_XY_VALUES | ROUND_XY_TO_GRID。所以参数是 int16（这就是 01 f4 00 00 占 4 字节而非 2 字节的原因），它们是一个 xy 偏移，且该偏移会取整到 网格。缩放位和 2×2 位都没置 ⇒ 变换是恒等变换——纯 平移，不缩放不旋转。

glyphIndex a4 67 = 0xa467 = 42087，是指向 同一字体内部的裸字形 ID；索引 42087 的轮廓是组合分音符（可读的名字 "dieresiscomb" 来自 post/cmap，不来自这些字节）。它像普通字形一样经 loca/glyf 解析——所以这种引用是递归的：组件本身 也可以是复合字形，层层嵌套，直到落到真正的轮廓上。

arg1、arg2 01 f4 00 00 = (0x01f4, 0x0000) = (500, 0)。因为 ARGS_ARE_XY_VALUES 置位，它们是一个摆放偏移， 绘制前加到组件的每个点上（把分音符右移 500 单位、上移 0）。 若那一位清零，同样这两个值就成了一对点匹配 ——“把已拼好部分的第 arg1 号点焊到本组件的第 arg2 号点上”—— 某些字体用这种替代锚定机制把附加符号吸到基字上。

没有 MORE_COMPONENTS 位（0x0020）⇒ 这是唯一的组件。整个字形就是“把 字形 42087 摆到偏移 (500, 0)，恒等变换”。复合字形的价值就在这里：中文 共享偏旁，带附加符号的拉丁字母共享基字加符号——轮廓只存一份，引用许多次——既省 空间，又保证每份拷贝在视觉上分毫不差。

不止一个组件

复合字形通常摆放好几个字形。没有计数字段——记录直接 首尾相接，每条记录用自己的 MORE_COMPONENTS 位（0x0020）表示“后面 还有一条”。解析器循环：读一条记录，该位置位就接着读下一条；最后一条记录 把它留空。（本教程的两款字体碰巧只用单组件复合字形，所以下面这串字节 是人为构造的例子——一个由基字 'a' + 分音符拼成的带音符 'ä'——但布局与真实情况分毫不差：）

每条记录各带各的 flags，所以组件可以混用不同的宽度和变换——一条用 int8 参数配恒等变换，下一条用 int16 参数带缩放。最后一条记录之后还可能出现一样 东西：只要有组件置了 WE_HAVE_INSTRUCTIONS（0x0100），一个 uint16 指令长度加相应数量的 hinting 字节就给这个字形收尾。否则 MORE_COMPONENTS 链一断，字形当场结束。

3.8编程题

用 pen 重放轮廓。用 RecordingPen 抓下 '─' （U+2500）和 '一'（U+4E00）的绘图命令。矩形的命令列表和带曲线那个 差在哪？曲线外点串由哪个命令携带？加分项： 对字形对象 dir() 一下，说出两个你还没用过的成员。

卡住了？给个方向（不给完整答案） gs = font.getGlyphSet(); gs[font.getBestCmap()[0x2500]].draw(pen) 会填好 pen.value（这个字形名叫 "boxdrawingslighthorizontal"）。直边 → lineTo；二次曲线段 → qCurveTo（隐含中点 fontTools 替你 重建好了）。
✓ 核对答案 '─' 全是 moveTo/lineTo×3/closePath（多边形，没有曲线）； '一' 含有 qCurveTo 命令，其中多出来的点就是曲线外控制点。值得一找的 成员：getCoordinates、recalcBounds、removeHinting、 drawPoints。

可按上面自查——或把两份命令列表贴给 AI 导师，让它核对。
