Module 04

Metrics & the em

Outlines say what a glyph looks like. Metrics say where it sits: how big the design grid is, how far the pen moves to the next glyph, and where the baseline and line edges fall. Get these numbers right and Latin, Han, and a terminal cursor all snap to the same grid; get them wrong and the same font renders at two different line heights on two machines.

concept

What makes text line up

A glyph's points (Module 3) are not in pixels. They live in abstract font units on a square grid called the em square. The side of that square is head.unitsPerEm — here 1000; 2048 is the other common value (TrueType fonts often pick a power of two). Nothing about the grid is tied to a display size:

The em is the unit of decoupling At a rendered size of N px, one em is exactly N px, and a font-unit coordinate c lands at c / unitsPerEm × N pixels. The designer works in a fixed 1000-unit world; the renderer scales that world to whatever size the user asks for. "Design once in abstract units, scale at draw time" is the same trick as vector graphics, DPI-independent UI, and SVG viewBoxes.

Horizontal metrics: advance & side bearings

Per-glyph horizontal metrics live in the hmtx table. Each glyph has two numbers:

  • advance width — how far the pen moves right from this glyph's origin to the next glyph's origin. This, not the ink, is what governs column position.
  • left side bearing (lsb) — the gap from the origin to the glyph's leftmost point. The right side bearing is implied: advance − lsb − blackboxWidth, so it isn't stored.

The baseline is y = 0. Ascenders go to +y, descenders to −y. The origin sits on the baseline at the left edge of the advance box.

The run-length trick in hmtx

The horizontal header hhea stores a count, numberOfHMetrics. The hmtx table then holds that many (advance, lsb) pairs — and after them, for every remaining glyph, it stores only an lsb. Those trailing glyphs all reuse the last advance in the array. So numberOfHMetrics can be smaller than numGlyphs.

Why this exists In a monospace or CJK font, thousands of glyphs share one identical advance. Storing it once and run-length-eliding the rest shrinks the table dramatically — a tiny version of the "don't repeat a constant" idea you see in PCM silence runs, sparse matrices, and columnar databases.

Vertical writing

For top-to-bottom CJK, the analogues are vhea / vmtx (both present in WenKai — you saw them in Module 1's directory). Instead of an advance width they carry an advance height: how far the pen drops to the next glyph when text runs in a vertical column. This connects to the vert feature in Module 6, which swaps in vertical glyph variants.

Line spacing: three competing ascent/descent pairs

Here is the famous trap. A font carries three sets of vertical line metrics, and applications disagree on which to honor:

  • hhea.ascent / descent / lineGap — the original typographic intent (Mac/CoreText lean on these).
  • OS/2 sTypoAscender / sTypoDescender / sTypoLineGap — the "typographic" set, used when the OS/2 fsSelection USE_TYPO_METRICS bit is set.
  • OS/2 usWinAscent / usWinDescent — the Windows clipping box; anything outside it may be chopped, and GDI uses it for line height.

When these three disagree, the same font file produces different line heights across platforms and apps. This is the single most common source of cross-platform line-height surprise.

Why the terminal grid lines up '一' advances 1000 = exactly 1 em: full-width. Every CJK ideograph in this font advances one full em, which is precisely why terminal columns align under Chinese text — each Han glyph occupies exactly one cell. The Latin 'A' and 'i' both advance 500 = half an em, so in this Mono face the Latin is half-width: two Latin cells equal one CJK cell. Note the side bearings differ (21 vs 73) — the ink sits in different spots inside the box — but it's the advance, not the lsb, that fixes the column. Same advance ⇒ same column position, whatever the ink does inside.

An app that sums hhea (1169 units) and one that sums sTypo (1000 units) will draw the same text at noticeably different line heights. That gap is the cross-platform line-height trap, in concrete numbers.

The run-length elision, concretely

Here numberOfHMetrics = 46821 but numGlyphs = 46830 — a difference of 9. So hmtx stores 46821 full (advance, lsb) pairs, then 9 bare lsb values. Those last 9 glyphs carry no advance of their own; each reuses the advance of glyph 46820 (the final entry in the pairs array). The font writer ordered the glyph table so that a run of glyphs sharing the final advance ends up at the very end, letting the table drop 9 redundant advances.

See it in build.py Both WenKai and ZhenKai advance '一' at 1000 / 1000 = 1.000 em. That equality is why lxgw-wenkai-zhenkai/build.py can drop the ZhenKai face in as the bold without re-metricizing anything: if both faces put every Han glyph one full em wide, CJK columns stay aligned after the swap. The build script prints this advance/unitsPerEm comparison as its sanity check before assembling the hybrid.

4.1concept

Units to pixels. Rendering at 16 px with unitsPerEm = 1000, how many pixels wide is the advance of '一' (advance 1000) versus 'A' (advance 500)?

Show solution Pixels = advance / unitsPerEm × N. For '一': 1000/1000 × 16 = 16 px (one full em). For 'A': 500/1000 × 16 = 8 px (half an em). Two 'A' cells (8+8) exactly equal one '一' cell (16).
4.2concept

Why the grid holds. Explain why every full-width CJK glyph advancing exactly 1 em makes a terminal grid line up, while the Latin at 500 still fits the same grid.

Show solution A monospace terminal lays out fixed-width cells. If every Han glyph advances exactly 1 em, each lands on a cell boundary and the next glyph starts at the next boundary — columns never drift. The Latin advances 0.5 em, an exact divisor of the cell, so two Latin glyphs fill one CJK cell and the grid stays intact. It works because 500 and 1000 are commensurate (one is a clean fraction of the other); an arbitrary advance like 537 would break alignment.

4.4concept

The shrinking table. Explain why numberOfHMetrics (46821) can be smaller than numGlyphs (46830). What do the trailing 9 glyphs use for their advance?

Show solution hmtx stores numberOfHMetrics full (advance, lsb) pairs, then only lsb values for the rest. The trailing 46830 − 46821 = 9 glyphs have no advance of their own; each reuses the last advance in the pairs array (that of glyph 46820). It's run-length elision: when a run of glyphs at the end share one advance, you store it once instead of 9 times, saving 9 × 2 bytes and signalling "same as last".
4.5concept

Three sources, one trap. Name the three ascent/descent sources, give this font's values from the walkthrough, and explain how two apps can render this same font at different line heights.

Show solution The three sources: (1) hhea.ascent/descent/lineGap = 928 / −241 / 0; (2) OS/2 sTypoAscender/Descender/LineGap = 880 / −120 / 0; (3) OS/2 usWinAscent/usWinDescent = 928 / 241. An app that derives line height from hhea (928 + 241 = 1169 units) draws taller lines than one using the typographic set (880 + 120 = 1000 units), and a Windows app may use the usWin clipping box instead. Because the three disagree and there's no universal rule for which wins (the USE_TYPO_METRICS bit only nudges some apps toward the sTypo set), the same font yields different line heights across platforms.
