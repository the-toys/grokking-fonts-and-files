模块 05

身份与家族

在任何应用里点一下加粗，字形就变了。这些字形从哪来？ 不是来自某个配置文件，而是来自烙在每个字体 name、OS/2、 head 表里的元数据。本模块先从真实的霞鹜文楷等宽 GB（LXGW WenKai Mono GB）里把这些元数据读出来，再拿它们办一件具体的事： 把两款毫不相干的字体——文楷与霞鹜臻楷 GB（LXGW ZhenKai GB）——焊成一个家族，让它的粗体是 真实绘制的第二款字：只改四张小表，不移动一个字形。

练习是穿插式的——每道题就跟在它考察的字段后面。概念题 附答案；编程题附可折叠的✓ 核对答案。

概念 · 1

家族是元数据，不是配置文件

在你的印象里，“一个字体家族”是一个整体，带几档字重。但落到磁盘上，它通常是 几个彼此独立的文件——每档字重/样式一个。 Regular.ttf 和 Bold.ttf 之间没有任何物理关联：没有清单文件，没有目录约定， 操作系统也不强制任何命名规则。那么，请求粗体时，系统是怎么摸到正确的那个文件的？

核心思想 字体家族不是配置文件，而是烙在每个文件 name、OS/2、head 表里的元数据。CoreText（macOS）、DirectWrite （Windows）和 fontconfig/FreeType（Linux）都会把文件归入同一个家族， 条件是这些文件共享同一个家族名，又各自声明不同的字重/样式。改元数据 就是改家族——一条轮廓都不用碰。

这为什么重要？因为一旦弄错，加粗按钮照样会做出某种效果 ——很丑的那种：

利害所在：伪粗体很丑 应用请求粗体时，如果家族里没有真正的粗体款，许多渲染器 会自己合成一个：用算法把 Regular 的轮廓加粗涂抹（embolden）或倾斜（oblique）。 得到的“伪粗体”/“伪斜体”，比起真正绘制的粗体，粗细不匀、效果粗糙。 提供一款真正绘制的粗体，并把标签写对、让操作系统找得到它，就是本模块的 全部任务。下面每个字段的存在，都是为了让你向文楷请求粗体时，操作系统交出臻楷 ——一款笔画确实更重、真实绘制的楷体。

所以计划很具体：拿两份互不相干的文件，只改必要的元数据，让操作系统看到 一个家族，里面有真 Regular 和真 Bold。接下来的四节，每节回答一个问题，顺势引出一个字段 ——靠什么把文件归成一组？哪一个是粗体？“粗体”存成什么样？字体 选择器怎么知道它是等宽的？——然后我们真刀真枪地做一遍改名重标。

概念 · 2

name 表：靠什么把两个文件归成一个家族

第一个问题：操作系统凭什么判定两个文件属于一家？答案 在 name 表里——一袋字符串。每条记录以四元组 (platformID, encodingID, languageID, nameID) 作键，因此同一个逻辑字符串可以按 多种编码、多种语言各存一份。nameID 说明这个字符串是什么含义：

nameID含义备注
1家族名传统家族名（RIBBI 分组）
2子家族名只能取 Regular / Bold / Italic / Bold Italic 之一——所谓 “RIBBI” 集合
3唯一 ID应当全局唯一
4全名供人阅读，如 "Family Bold"
6PostScript 名ASCII、不含空格，必须全局唯一
16 / 17排印家族名 / 子家族名存在时覆盖 1/2；用于样式多于 4 种的家族

分组机制很简单：nameID 1（家族名）相同，nameID 2 （子家族名）不同。给两个文件写上一模一样的家族字符串，一个标 "Regular"、另一个标 "Bold"，操作系统就会把它们归档成一个家族、两个字款。这个共享的家族名，就是我们 要用的粘合剂。

有个坑值得现在就记下，因为它在最后会咬我们一口。1/2 这对字段只能表达 RIBBI 四种样式，因为 2 被限定在那个集合里。带 Light、Medium、SemiBold…… 的家族 得靠 16/17：真名自由地写在 16/17 里，1/2 则把一切折叠到 Regular/Bold 上， 安抚老应用。16/17 一旦存在就以它们为准——所以只要任一源字体 带着它们，就会盖过我们写的 1/2，把两个文件重新拆开。因此“删掉 16/17”是合并流程里 正式的一步，不是可有可无的收尾。

概念 · 3

OS/2.usWeightClass：“加粗”选中的那个数字

现在操作系统看到了一个家族、两个字款。第二个问题：用户点加粗时，系统在 两者之间挑哪个？凭一个数字。字重就是 OS/2.usWeightClass 里的一个普通整数：400 = Regular，700 = Bold（范围 100–900）。应用“请求粗体”时，底层实际是在向家族索要字重等级为 700 的那一款。这个数字写错，加粗按钮就会选中错误的文件——或者什么都选不到。

5.3概念题

用你自己的话讲讲家族粘合剂。负责分组的两个字段你已经都认识了。 用两三句话解释：两个物理上分离的字体文件，是怎么进到同一个“家族”并且 加粗可用的——并点名起作用的具体元数据。

看答案 两个文件带相同的 nameID 1 家族字符串，但 nameID 2 子家族不同（"Regular" vs "Bold"）、OS/2.usWeightClass 不同（400 vs 700），且 fsSelection/macStyle 样式位与之匹配。操作系统的字体系统 （CoreText / DirectWrite / fontconfig）按家族名把文件分组，并把“粗体”请求 路由到声明字重 700 的那一款——全程不涉及任何配置文件。
概念 · 4

样式位：存两份，还要互相校验

“这是粗体”这件事，不止记在子家族名和字重数字里。粗体/斜体 的事实还会以位标志的形式，冗余地记录在两张表里，渲染器会拿两处 互相校验。两处必须一致：

字段位值含义
OS/2.fsSelection00x01ITALIC
OS/2.fsSelection50x20BOLD
OS/2.fsSelection60x40REGULAR
head.macStyle00x01BOLD
head.macStyle10x02ITALIC

如果 fsSelection 说 BOLD 而 macStyle 说不是粗体，这款字就 自相矛盾了：不同平台读不同的表，对这个文件是什么会得出不同结论。所以我们的 改名重标必须把两处都写上，而且写得一致。

5.1概念题

解码选择位。文楷的 OS/2.fsSelection 是 0b1000000。置位的是第几位？含义是什么？换成 0b0100000 又是什么意思？

看答案 0b1000000 = 第 6 位置位 = REGULAR（0x40）——文楷作为 Regular 款所带的值。0b0100000 = 第 5 位置位 = BOLD （0x20）——合并时我们给臻楷设的值。（第 0 位 = ITALIC，0x01。）
5.2概念题

一件事记两处。为什么这个格式要把粗体标志 同时存进 OS/2.fsSelection 和 head.macStyle？两处不一致 有什么风险？

看答案 历史层叠所致：head.macStyle 是早年 Apple/TrueType 的样式字段， OS/2.fsSelection 是后来 Windows/OpenType 的。两者都留存至今，不同 渲染器/平台各读各的。一旦不一致，这款字就自相矛盾 ——一个平台当它是粗体，另一个当它是常规体，于是它可能被选进错误的槽位， 或被套上伪样式。这正是合并时要把两处写一致的原因。
概念 · 5

PANOSE 与等宽标志

身份还剩最后一块，因为这是个等宽家族，而字体选择器支持按等宽 筛选。选择器不会去量字形——它读的是一份自我声明。OS/2.panose 是 10 个字节的粗粒度视觉分类（衬线风格、字重、比例……）。这里要紧的是索引 3 的那个字节，bProportion：9=Monospaced（等宽），3=Modern （比例宽度）。应用判断“这是等宽吗？”还会看另一个标志：post.isFixedPitch。 注意两者都只是声明——它们描述字体，并不能让字体变成等宽 （真正决定等宽的是前进宽度，模块 4）。改名重标必须把它们设置好，混合家族才能继续 出现在等宽筛选之下。

解读 · 6

需要的字段就这么多。下面是文楷的原始身份，直接从 LXGWWenKaiMonoGB-Regular.ttf 里读出——每一行都对应上面几节讲过的一个字段：

5.4编程题

亲手读出身份。改动任何东西之前，先确认初始状态。 用 fontTools 打开文楷，打印 usWeightClass、bin(fsSelection)、 bin(head.macStyle) 和 name ID 1/2/6，与上面两张表对一对。

# python this.py LXGWWenKaiMonoGB-Regular.ttf
import sys
from fontTools.ttLib import TTFont
f = TTFont(sys.argv[1])
os2, head, name = f["OS/2"], f["head"], f["name"]
print("usWeightClass", os2.usWeightClass)
print("fsSelection  ", bin(os2.fsSelection))
print("macStyle     ", bin(head.macStyle))
for i in (1, 2, 6):
    print("name", i, repr(name.getDebugName(i)))

把你的输出贴给 AI 导师，让它核对 usWeightClass=400、fsSelection=0b1000000、 macStyle=0b0，以及三条 name 字符串是否与表格一致。

现在开始改名重标。我们手上有两份互不相干的文件：文楷（笔画轻、带书法味的楷体）和 臻楷（笔画更重的楷体）。目标是一个家族 "LXGW WenKai ZhenKai Mono GB"，其中 Regular = 文楷、Bold = 臻楷。我们会写一个小函数——就叫 build_face()—— 给每个文件重新打标签，让操作系统把它们归成一组。每一步写入的都是你刚认识的一个字段：

1 · 家族相同，子家族不同 （概念 2）

两个文件的 nameID 1 写入一模一样的 FAMILY 字符串， nameID 2 则各不相同：

set_name(font, NAME_FAMILY, family)      # id 1 — SAME on both files
set_name(font, NAME_SUBFAMILY, style)    # id 2 — "Regular" vs "Bold"

这个共享的家族名就是粘合剂。set_name() 会把每条记录写给 两个要紧的平台——Windows (3,1,0x409) 和 Mac (1,0,0)——写之前先调 removeNames() 清掉同 ID 的 旧记录，确保没有原始字符串残留（这也是 value 必须是 ASCII 的原因： Mac 记录才编码得了它）。

2 · 字重数字 （概念 3）

os2.usWeightClass = weight   # 400 for WenKai, 700 for ZhenKai

加粗按钮解析的，说白了就是这个数字：向 FAMILY 索要字重 700， 递到你手上的就是臻楷。

3 · 样式位，两处都写 （概念 4）

先清掉三个互斥的选择位，再只置上其中一个， 并把粗体位镜像到 macStyle，让两张表保持一致：

os2.fsSelection &= ~(FS_ITALIC | FS_BOLD | FS_REGULAR)
os2.fsSelection |= FS_BOLD if bold else FS_REGULAR   # 0x20 or 0x40
head.macStyle  &= ~(MAC_BOLD | MAC_ITALIC)
if bold:
    head.macStyle |= MAC_BOLD                        # 0x01

结果：臻楷的 fsSelection 置第 5 位（BOLD，0x20）、macStyle 置第 0 位（BOLD，0x01）；文楷保持 fsSelection 第 6 位（REGULAR，0x40）、 macStyle = 0。在每个平台上都自洽。

成果 现在，任何用 "LXGW WenKai ZhenKai Mono GB" 渲染并请求粗体的应用，拿到的都是 臻楷——真实绘制的粗笔画。没有伪粗体的涂抹，也不用逐应用配置。 这相当于 Linux fontconfig 里 <alias>“把 臻楷当作文楷粗体”那条规则的跨平台版本——只不过是靠直接改字体而非写规则文件实现的， 所以在 macOS 和 Windows 上同样有效。

4 · 声明等宽，不动一个字形 （概念 5）

post.isFixedPitch = 1 if monospace else 0
if getattr(os2, "panose", None) is not None:
    os2.panose.bProportion = 9 if monospace else 3   # 9=Mono, 3=Modern

这一步只翻转字体选择器用来筛选的那两个标志，不移动 任何前进宽度——也不需要移，因为 CJK 字形本来就是全角的 （前进宽度 == 1 em，模块 4）。这条等式我们在模块 4 里已经证过：臻楷的 一 前进 1000/1000 em，与文楷完全一致，所以粗体 CJK 列能和常规体 对齐，无需重新调整度量。

5 · 给每一款独一无二的名字——3 与 6 让混合家族不与原版相撞

ps_name = f"{family.replace(' ', '')}-{style}"   # e.g. LXGWWenKaiZhenKaiMonoGB-Bold
set_name(font, NAME_UNIQUE, ps_name)             # id 3 — Unique ID
set_name(font, NAME_PS, ps_name)                 # id 6 — PostScript name (ASCII, no spaces)

nameID 6（PostScript 名）和 3（唯一 ID）按规范应当 全局唯一——macOS 的 CoreText 就拿它们当字体的身份键。你的机器上 多半还装着原版的文楷和臻楷，所以如果重标后的 Regular 成员仍沿用 文楷的原名 "LXGWWenKaiMonoGB-Regular"，就会有两个文件主张同一个身份。 CoreText 对每个 PostScript 名只保留一款字体，会做去重——留下的可能是原版， 而不是我们的。届时对我们家族的请求会悄悄解析到错误的文件：混合家族 渲染不出来，或者 Regular↔Bold 的关联断掉，因为它需要的成员被遮蔽了。把每个 名字都从新家族名派生，就给了每一款独立的身份，与原版和平共处。

6 · 删掉排印名——16 与 17 让简单的 RIBBI 名生效

font["name"].removeNames(nameID=NAME_TYPO_FAMILY)     # drop id 16
font["name"].removeNames(nameID=NAME_TYPO_SUBFAMILY)  # drop id 17

回想概念 2：16/17 一旦存在就覆盖 1/2。只要任一源字体 还带着它们，操作系统读到的就是各文件原来的排印家族——两个 文件立刻被拆回各自的家族，我们刚搭好的 Regular+Bold 分组随即报废。 剥掉 16/17，操作系统就只能落到我们简单的 RIBBI（1/2）名字上。

5.5概念题

为什么要唯一 ID？为什么 nameID 3（唯一 ID）和 nameID 6（PostScript） 必须在各别名之间互不相同，并且也不同于原版？两款字共用它们时， macOS 上会出什么 bug？

看答案 CoreText 在一定程度上靠 ID 3/6 识别一款字。两个文件共用时，CoreText 可能把 其中一个对另一个去重或遮蔽——当成同一款字——在请求该家族时端上 错误的文件（属于字体优先级/遮蔽类 bug）。给每个别名一个唯一的 family-style 形式的 PostScript 名/唯一 ID，每一款就都保持独立， 系统交付的才是混合家族，而不是仍然装在机器上的原版。
概念 · 7

改名重标刻意不碰什么

上面所有改动只落在四张表里：name、OS/2、head、 post。重量级的表——glyf、cmap、hmtx——保持 逐字节不变。这正是模块 1 那一课兑现的时刻：表是彼此独立的 数据块，所以你重写几 KB 的元数据，24 MB 的轮廓原封不动。这 也是共享表 .ttc 得以成立的前提：每个 Regular 款的 glyf 字节完全 相同（每个 Bold 款亦然），集合写入器就能把这份轮廓数据只存一份——为模块 7 埋下伏笔。

自己动手验证 specimens/extract.py LXGWWenKaiMonoGB-Regular.ttf 会打印出上面展示的 usWeightClass、fsSelection、macStyle、PANOSE 和 name 记录——即改名重标的输入。把上述六步 套用到每个文件的副本上，你就亲手组装出了这个家族；模块 7 再把 重标后的各款打包成一个 .ttc。
5.6概念题

迷你综合题：表的清单。准确列出合并会改哪些表、 刻意不碰哪些表。不碰 glyf/hmtx 为什么能保住终端里的 列对齐（模块 4）？它又是怎么让共享表 TTC（模块 7）成为可能的？

看答案 改动的：name（ID 1,2,3,4,6；删除 16/17）、OS/2 （usWeightClass、fsSelection、panose.bProportion）、head（macStyle）、 post（isFixedPitch）。不碰的：glyf、cmap、 hmtx、loca 及其余全部。不动 hmtx/glyf 意味着前进宽度不变——CJK 保持全角（前进宽度 == 1 em），终端里粗体列 与常规体对齐，无需重新调整度量。又因为每个 Regular 款共享 逐字节相同的 glyf（每个 Bold 款亦然），.ttc 写入器就能把 这份重量级数据只存一份（靠 shareTables）——集合的体积约等于两份源文件之和，而不是 9 倍。 这就是模块 7 的内容。
