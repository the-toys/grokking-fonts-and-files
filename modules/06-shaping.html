Module 06

Shaping & layout

You have a font that maps characters to glyphs (Module 2), draws their outlines (Module 3), and knows how wide each one is (Module 4). But real typography is more than "one character, one glyph, placed side by side". Shaping is the step that rewrites and re-positions the glyph run before anything is drawn — turning f+i into a single fi, nudging AV closer together, or swapping a glyph for its sideways form so CJK text can flow top-to-bottom.

concept

How glyphs combine and move

Recall the pipeline so far. Text arrives as Unicode codepoints. The cmap table (Module 2) turns each codepoint into a glyph ID, giving you a first, naïve run of glyph IDs. Then the outlines (Module 3) are drawn at the advance widths from hmtx (Module 4). The missing middle step is shaping:

text (Unicode)
   │  cmap            ← codepoint → glyph-ID  (Module 2)
   ▼
glyph-ID run (naïve)
   │  SHAPING         ← GSUB rewrites + GPOS repositions   ◀── this module
   ▼
final positioned glyph run
   │  outline fill at advances  (Modules 3 + 4)
   ▼
pixels

Shaping sits between cmap and final placement. It is driven by two tables, a substitution table and a positioning table, that almost always travel together:

  • GSUB — Glyph SUBstitution. Replaces glyph IDs with other glyph IDs. This is where ligatures (f+i → fi), contextual alternates, composition / decomposition, fraction building (1/2 → ½), super-/subscripts, stylistic alternates, and vertical-form swaps live. The count of glyphs can change.
  • GPOS — Glyph POSitioning. Moves glyph IDs without changing which glyphs they are. Kerning, mark-to-base and mark-to-mark attachment (accents), cursive joining, and vertical-metric adjustments all happen here. The glyph IDs are untouched; only their x/y offsets and advances shift.

Both tables share the same internal hierarchy:

Script  (e.g. latn, hani)
  └─ LangSys  (language system, often just the default)
       └─ Features  (4-letter tags: 'liga', 'kern', 'calt', 'vert' …)
            └─ Lookups  (the actual substitution / positioning rules, run in order)

A feature is a four-letter tag that an application chooses to turn on; the tag is just a name that points at one or more lookups, which hold the real rules and execute in sequence. Some features are default-on (the engine applies them unless asked not to — e.g. ccmp, calt); others are opt-in, applied only when the layout requests them (e.g. frac, sups).

Who actually runs the rules A font only stores GSUB/GPOS; a shaping engine executes them. HarfBuzz is the de-facto open-source one — it powers Chrome, Firefox, Android, and LibreOffice. The OS equivalents are CoreText (macOS / iOS) and DirectWrite (Windows). They read the script + features, walk the lookups, and emit the final positioned glyph run.

For Latin, skipping shaping is ugly but readable. For complex scripts it is not optional: Arabic letters must be joined and take initial/medial/final/isolated forms, and many Indic scripts reorder vowels and form conjuncts. There, the raw cmap output — one glyph per codepoint, left to right — is genuinely unreadable. Shaping is what makes the text correct, not merely prettier.

Two "maps" that are easy to confuse cmap is codepoint → glyph: one lookup, no context, the entry point. GSUB is glyph → glyph(s) in context: typography, applied after. And beware a second collision: GSUB's ccmp "composition" is a shaping substitution (swap these glyph IDs for that one), which is a different thing entirely from the composite glyphs of Module 3 (one outline reusing another outline as a component). Same word, different layer.

Why this exact feature mix exists The font ships GSUB vert together with GPOS vert, vhal, and halt for one concrete reason: CJK text can be set top-to-bottom. When a layout engine switches to vertical mode, GSUB vert swaps glyphs to their vertical forms (e.g. rotated brackets and punctuation), while GPOS vert / vhal / halt adjust the metrics for the new orientation. That positioning side leans on the vhea and vmtx tables from Module 4 — the vertical header and vertical advances. Substitution and positioning working in tandem is exactly the GSUB + GPOS division of labour, and here it is in service of vertical CJK.

Feature tags are abstract until you watch them rewrite a run. Every row below is a real substitution from this font, produced by shaping the string through HarfBuzz and diffing the glyph run against the naïve cmap output. Read each as what the glyph IDs were → what GSUB turned them into. Watch the count change — that's the GSUB signature.

The .full / .nut / .salt suffixes are just glyph names — an alternate glyph the font ships specifically for that feature. GSUB's whole job is choosing which glyph name each position resolves to.

GSUB substitution types — the four shapes a rule can take Every substitution above is one of a handful of lookup types: single (1→1, e.g. sups, vert, salt), multiple (1→many, the ccmp dash decompose), ligature (many→1, frac and the ccmp À compose), and contextual / chaining (swap only when neighbours match, which is how calt and most script logic work). WenKai's 12 GSUB lookups are exactly these types (single ×5, multiple, ligature ×3, chaining ×2).

Beyond what this font ships, the same machinery powers the classics — worth knowing even though WenKai doesn't declare them:

  • liga — f+i → a single fi glyph, so the dotless collision is drawn as one shape (many→1, like frac).
  • Arabic joining — one letter becomes four glyphs by position (isolated / initial / medial / final); the shaper picks the contextual form so letters connect.
  • Indic reordering — a vowel typed after a consonant is substituted and moved before it, and consonant clusters fuse into conjunct glyphs. Mandatory, not cosmetic.

GPOS in action: same glyphs, moved

GPOS never changes which glyphs you have — the run length is fixed. It changes their x/y offset and advance. WenKai's GPOS is all about vertical CJK metrics; here is a real position diff from HarfBuzz (horizontal vs. vertical setting of the same character):

featureglyphhorizontalverticalwhat happened
vert / vhal。x-advance 1000, no offsety-advance 1000 (downward), y-offset −880the small stop is lifted to the top-right corner of its em cell and the pen now moves down the column, not right

And the GPOS classics the wider world relies on (again, not in WenKai, but the same table):

  • kern — AV: leave both glyphs as-is but shift the V left by ~80 units so the shapes tuck together. Pure positioning, zero substitution.
  • mark (mark-to-base) — place a combining accent over its base by an anchor point (centered on the letter), not by advance width, so e+◌̈ lands the dieresis dead-center regardless of the e's width.
  • mkmk (mark-to-mark) — stack a second accent above the first (e.g. Vietnamese), spacing each off the previous mark.
  • curs (cursive attachment) — weld the exit anchor of one glyph to the entry anchor of the next, so an Arabic joining stroke flows unbroken.
The one-line test: did the count change? If a run got longer or shorter, that was GSUB (ligature, decompose, fraction). If the run is the same glyphs but sitting in different places, that was GPOS (kern, mark attach, vertical metrics). GSUB decides which glyphs; GPOS decides where — always in that order.

6.1concept

Order the pipeline. Put these four stages in the order they run, and say in one phrase what each does: GPOS, cmap, GSUB, outline fill.

Show solution 1. cmap — turn each Unicode codepoint into a base glyph ID. 2. GSUB — substitute glyph IDs (ligatures, alternates, vertical forms): the run can change length. 3. GPOS — move the glyphs (kern, attach marks, adjust metrics) without changing which glyphs they are. 4. outline fill — draw each final glyph's outline at its placed position / advance. GSUB before GPOS: you decide which glyphs first, then where they go.
6.2concept

GSUB or GPOS? For each tag — calt, frac, vert — say which table it belongs to and what visible effect it produces.

Show solution calt = GSUB — replaces a glyph based on its neighbours (contextual alternate shapes). frac = GSUB — replaces a digit/slash/digit run with a single fraction glyph, e.g. 1/2 → ½. vert appears in both here: GSUB vert swaps a glyph for its vertical form, and GPOS vert adjusts that glyph's position/metrics for vertical setting. The two calt/frac change which glyph; vert changes the glyph and nudges its placement.
6.3concept

cmap vs GSUB. What is the difference between a cmap mapping and a GSUB substitution? Give one concrete example of each.

Show solution cmap is a single, context-free table that maps one codepoint to one base glyph — e.g. U+4E00 '一' → glyph u4E00. GSUB takes the resulting glyph run and rewrites it in context, possibly changing the count — e.g. the glyphs for f+i → a single fi ligature, or a digit run under frac collapsing to ½. cmap answers "what glyph does this character start as?"; GSUB answers "given the surrounding glyphs and the requested features, what should this become?".

6.5concept

Why some scripts can't skip shaping. Plain English text mostly survives on raw cmap output, but Arabic and Indic scripts genuinely require a shaping engine. Why?

Show solution In Latin, codepoints already sit in visual order and each letter has essentially one form, so one-glyph-per-codepoint is readable (you only lose ligatures/kerning — cosmetic). Arabic letters join and change shape by position (initial / medial / final / isolated), so the engine must pick the right contextual form and connect them. Many Indic scripts reorder — a vowel typed after a consonant may render before it — and combine consonants into conjunct glyphs. Those are mandatory substitutions and reorderings the shaping engine performs via GSUB/GPOS; without it the raw cmap run is in the wrong order and the wrong shapes — unreadable, not just unpolished.
6.6concept

Two kinds of "composite". GSUB ccmp is called glyph composition / decomposition, and Module 3 had composite glyphs. Both "combine" things. What is actually different?

Show solution Different layers. GSUB ccmp is a shaping substitution: it rewrites the glyph-ID run — e.g. splitting a precomposed character into base + mark glyphs (or recombining them) so later mark-positioning lookups can attach them cleanly. It operates on the sequence of glyphs, before drawing. Module 3's composite glyphs are an outline-reuse mechanism inside glyf: one glyph's outline is defined by referencing another glyph as a component placed at an offset (e.g. dieresis reusing dieresiscomb at x=500). That's about how a single glyph's shape is stored, not about rewriting the run. ccmp changes which glyphs; glyf composites change how one glyph is drawn.
