Module 01

The sfnt container

A font file is a binary container: a small directory at the front saying "here's what's inside and at what byte offset", followed by the data blobs (called tables) it points to. Learn this skeleton once and you can read fonts, and ZIP, ELF, PNG, TIFF — they all share it.

Exercises are interleaved — each sits under the concept it tests. Concept ones have a solution; programming ones have a collapsible check.

concept

The one big idea: index + payload, joined by offsets

Open a .ttf / .otf and you are looking at an sfnt ("spline font" / "scalable font") container. Its shape is universal to random-access binary formats:

  • A header (12 bytes) announcing the format and how many tables follow.
  • A table directory: one fixed-size record per table — its 4-byte tag, a checksum, its offset from the start of the file, and its length.
  • The table bodies themselves, sitting anywhere in the file, in any order, reachable only through the offsets in the directory.
The transferable insight The directory is sorted one way (by tag, for fast lookup) while the payload is laid out another way (whatever order the writer chose). The offset field is the only thing linking the two. "Index sorted for search, payload placed for convenience, joined by absolute offsets" is the essence of every seekable binary format.

Two consequences worth internalizing now:

  • You never parse a font front-to-back. You read the tiny directory, then seek to whichever table you need. A renderer that only wants character mapping reads the header, finds cmap in the directory, jumps to its offset, and ignores the 24 MB of outlines.
  • Tables are independent blobs. Editing one (say, renaming the font) doesn't require touching any other — exactly what makes the retag trick in Module 5 cheap.

Magic numbers & tags

The first 4 bytes are a magic number — how a file format announces "this is what I am". For sfnt the values you'll meet:

first 4 bytesmeaning
00 01 00 00TrueType outlines (a glyf table) — version 1.0
4F 54 54 4F ("OTTO")CFF / PostScript outlines
74 74 63 66 ("ttcf")a font collection (Module 7)
74 72 75 65 ("true")legacy Apple TrueType

A tag is just four ASCII bytes used as a name — 'G','P','O','S' is the integer 0x47504F53. Tags are everywhere: PNG chunks, RIFF/WAV, Mac type codes. Note the quirk that one table is literally named OS/2 (with the slash) — still 4 bytes.

walkthrough

Decoding the real header of WenKai

Here are the genuine first 12 bytes of LXGWWenKaiMonoGB-Regular.ttf (everything in OpenType is big-endian — most-significant byte first, the opposite of x86 memory order):

00 01 00 00   00 11   01 00   00 04   00 10
└────┬────┘   └─┬─┘   └─┬─┘   └─┬─┘   └─┬─┘
 sfntVersion  numTab  srchRng entSel  rngShift
bytesfieldvaluemeaning
00 01 00 00sfntVersion0x00010000TrueType outlines inside
00 11numTables1717 directory records follow
01 00searchRange256binary-search hint
00 04entrySelector4binary-search hint
00 10rangeShift16binary-search hint

Those last three are a fossil of 1990s optimization: they let a reader binary-search the directory without computing logarithms at runtime. They're fully derivable from numTables = 17:

largest power of two ≤ 17   = 16 = 2⁴
searchRange   = 16 × 16 bytes/record = 256   ✓
entrySelector = log₂(16)             = 4     ✓
rangeShift    = 17×16 − 256          = 16    ✓

Modern parsers ignore these and just loop, but they're a tell that the directory is sorted by tag (otherwise binary search would be impossible). A malformed font can lie here; robust readers recompute.

See it for yourself xxd -l 64 LXGWWenKaiMonoGB-Regular.ttf shows the header + first records. specimens/extract.py LXGWWenKaiMonoGB-Regular.ttf prints this whole directory.

Look at the offsets: head physically comes first (284), glyf is a giant block in the middle, GPOS sits near the very end (25.7 MB in). Yet in the on-disk directory the records appear as GPOS, GSUB, OS/2, cmap… — alphabetical by tag (uppercase ASCII sorts before lowercase, so G=0x47 precedes c=0x63). That gap between record order and body order is the whole lesson.

Decode a directory record cold. The OS/2 record's raw bytes are:

Show solution offset = 0x00000198 = 408; length = 0x00000060 = 96. Matches the OS/2 408 96 row. (Tag 4f 53 2f 32 = "OS/2", fe 89 fd aa is the checksum.)
1.4concept

Payload order ≠ directory order. cmap is at offset 187808, length 217212. Compute the byte where it ends. Which table's body begins there? Is that table cmap's alphabetical neighbor in the directory? What does the answer prove?

Show solution 187808 + 217212 = 405020, which is exactly where prep begins. But alphabetically cmap's directory neighbors are OS/2 and gasp — not prep. So the bodies are laid out independently of the sorted record order: the offset is the only link. (Tables here happen to be tightly packed end-to-end, but that isn't required — there can be padding/gaps.)

Integrity: checksums

Each record carries a checksum — the table's bytes summed as big-endian uint32s with 32-bit wraparound. The head table additionally stores checkSumAdjustment, a fudge value chosen so the whole file checksums to the constant 0xB1B0AFBA. Same job as a CRC in PNG/ZIP: detect corruption, not tampering (that's what the DSIG digital-signature table is for).

Required vs optional tables A valid TrueType font must contain cmap, head, hhea, hmtx, maxp, name, OS/2, post plus glyf+loca (for TrueType outlines). Everything else (GPOS, GSUB, gasp, prep, meta, vhea, vmtx…) is optional. WenKai carries the vertical ones because CJK text can be set top-to-bottom.

1.6concept

Synthesize. In one or two sentences each: (a) why can a text renderer that only needs character→glyph mapping skip reading 24 MB of this 25 MB file? (b) why does renaming a font (Module 5) not require rewriting the glyf table?

Show solution (a) It reads the 12-byte header + directory, finds the cmap record, and seeks straight to offset 187808 — random access means it never touches glyf at 592352. (b) A font's identity lives in small metadata tables — its names in name, its weight/style in OS/2 and head's macStyle flags — while the outlines live in glyf. (You'll dissect those metadata tables in Module 5.) Tables are independent blobs, so you rewrite the small metadata tables and leave the giant outline table byte-for-byte identical — which is exactly why the hybrid .ttc can re-label fonts almost for free.
