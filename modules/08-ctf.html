Bonus round

Two font CTF challenges

The text you see on screen is the end of a pipeline: code points go through cmap to glyph IDs (Module 2), glyph IDs through glyf to outlines (Module 3), and the whole run gets rewritten by shaping (Module 6) before a single pixel is drawn. Every one of those stages is a place to hide meaning — which is exactly why "font" is a recurring capture-the-flag genre. Here are two classic archetypes, each shipping a real, tiny font you can download and crack with the tools this course already taught you.

Both fonts are subset from the specimen font and rebuilt by specimens/ctf/build_ctf.py, which self-verifies (recovers each flag before writing the file). Nothing here is decoration — the flags are genuinely in the bytes. Flags are flag{…}.

The one idea behind both What a font renders and what its bytes literally say are two different things, joined by a lookup you can read. A challenge hides the flag in the gap: challenge 1 puts it in the cmap (the character→glyph lookup), challenge 2 puts it in GSUB (the glyph→glyph rewrite). Solving means reading the lookup instead of trusting the render.
challenge · cmap

1 · The rendered text lies

A scraper pulled some text off a page and got back a run of "tofu" — code points in the Private-Use Area (U+E000, U+E001, …) that show as empty boxes in any normal font. But the page shipped its own font, and rendered with that font the same code points spelled out a flag. You have the font. Recover the flag.

cmap_swap.ttf leaked.txt

leaked.txt is the exact decoy string the scraper captured: the code points U+E000 … U+E017, in order. Rendering it with cmap_swap.ttf displays the flag — but you can recover it without rendering anything at all.

C1cmap

Find the flag. This is exactly the anti-scraping trick real sites use (猫眼, 起点, and friends): the HTML carries meaningless code points, and a bundled font's cmap maps them to the glyphs that draw the real characters. So copying the text yields garbage, but looking at it reads fine. Invert the map and read the flag off the glyphs.

Hint From Module 2: font.getBestCmap() gives you {codepoint: glyphName}. This font's cmap contains only the Private-Use code points — nothing maps ASCII. The glyph names are still intact (they live in post, "just for humans" — Module 2), so a glyph named braceleft draws {. Walk the code points in sorted order and translate each glyph name back to its character.
Show solution

The whole solve is one cmap inversion — no rendering, no font library beyond fontTools:

from fontTools.ttLib import TTFont
from fontTools.agl import toUnicode      # glyph name -> character

f = TTFont("cmap_swap.ttf")
cmap = f.getBestCmap()                 # {codepoint: glyphName}, Private-Use only
flag = "".join(toUnicode(cmap[cp]) for cp in sorted(cmap))
print(flag)

The cmap holds U+E000→f, U+E001→l, U+E002→a, U+E003→g, U+E004→braceleft, … Sorting by code point recovers the author's order (that ordering is the only place the sequence is stored — the glyphs themselves sit in the file in plain GID order, so dumping post alone gives you the letter set but not the flag). Output:

flag{c0d3po1nts_can_l13}

Why "cmap can lie" (Module 2, made literal): the cmap is a lookup, not an identity — the byte U+E000 has no inherent glyph, only the one this table assigns it. A hardened version would strip the post names (set it to format 3.0), forcing you to render each mapped glyph and read the outlines instead (Module 3) — the same idea, one rung harder.

challenge · GSUB

2 · The flag that only exists while shaping

This font's cmap is honest — dump it and every character maps to itself, @ to an at-sign, a to an a. There is no flag in the cmap, none in the glyph names, nothing for strings to find. Yet a banner on the site turned a lone @ into a whole flag on screen. Where did it come from?

gsub_reveal.ttf

The flag is assembled at shaping time — the step between "characters" and "final glyphs" (Module 6). It exists in the rendered run but never as stored text.

C2GSUB

Find the flag. Recall the pipeline from Module 6: cmap → GSUB (rewrite) → GPOS (reposition) → draw. A GSUB rule can do what a cmap physically cannot — turn one glyph into many (a multiple substitution). Here one calt lookup expands the seed glyph @ into the full flag glyph run. Read the lookup, or just shape the seed.

Hint Two ways in. Static: dump the layout table — ttx -t GSUB gsub_reveal.ttf, or walk it in fontTools — and look for a MultipleSubst (LookupType 2). Its .mapping is {inputGlyph: [outputGlyphs]}. Dynamic: shape the string "@" with HarfBuzz (Module 6's hb.shape) and read out the resulting glyph names.
Show solution

Static — read the substitution straight out of GSUB:

from fontTools.ttLib import TTFont
from fontTools.agl import toUnicode

f = TTFont("gsub_reveal.ttf")
for lookup in f["GSUB"].table.LookupList.Lookup:
    for sub in lookup.SubTable:
        if getattr(sub, "mapping", None):        # MultipleSubst (type 2)
            for src, dst in sub.mapping.items():
                print(src, "->", "".join(toUnicode(g) for g in dst))

prints at -> flag{gsub_1s_a_r3wr1t3r}. The @ glyph is mapped to the 24-glyph sequence [f, l, a, g, braceleft, g, s, u, b, …, braceright].

Dynamic — let the shaper do it (Module 6):

import uharfbuzz as hb
face = hb.Face(open("gsub_reveal.ttf", "rb").read())
buf = hb.Buffer(); buf.add_str("@"); buf.guess_segment_properties()
hb.shape(hb.Font(face), buf, {"calt": True})
order = TTFont("gsub_reveal.ttf").getGlyphOrder()
print("".join(order[i.codepoint] for i in buf.glyph_infos))

One input cluster shapes into 24 glyphs. The flag:

flag{gsub_1s_a_r3wr1t3r}

Why the cmap couldn't do this (Module 2 vs 6): a cmap maps each code point to exactly one glyph — it cannot fan one input out into a run, and it cannot depend on context. Rewriting a glyph stream is GSUB's job, and it happens after the cmap, only while shaping. That is why the flag is invisible to strings, to the cmap, and to the glyph names — it is not stored anywhere, it is computed.

Reproduce / harden ~/.venv/bin/python specimens/ctf/build_ctf.py rebuilds both fonts from the specimen and re-verifies each flag before writing it (prints PASS/PASS). Want them harder? Strip post names so challenge 1 forces outline rendering (Module 3); swap challenge 2's plain multiple-substitution for a chained-context lookup so the flag only fires next to a trigger you must first discover in the ScriptList/FeatureList.
The other classic you now have the tools for A third common archetype hides the flag in the outlines: ordinary-looking cmap, but glyphs whose curves draw the flag characters, mapped to scattered code points so they never appear in normal text — you dump or render every glyph to find them (pure Module 3). Same principle as these two: trust the bytes, not the render.
