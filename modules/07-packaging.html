Module 07

Packaging & frontiers

You can now read a font cover to cover. The last question is how to ship it: how one file can carry many faces (collections), infinitely many faces (variable fonts), the same faces in a smaller download (web fonts), and finally how the resolution-independent outline you studied in Module 3 turns into actual pixels. Everything here reuses the container model from Module 1 — nothing new is invented at the bottom.

concept

Shipping fonts: collections, variable, web, and rasterization

A single .ttf/.otf holds one face. Real-world delivery almost always needs more than one face, or a smaller file, or both. There are four largely orthogonal answers, and each is just a different arrangement of the same sfnt tables.

1 · Font collections (.ttc / .otc)

Recall the magic numbers from Module 1: a collection file starts with 74 74 63 66 = "ttcf". Instead of one sfnt directory, the file begins with a collection header listing byte offsets to several embedded sfnt table-directories. The crucial trick: two directories can have records whose offset fields point at the same physical table bytes.

The transferable insight A collection is deduplication by shared offset. If N faces share their outlines, the heavy tables (glyf, cmap, hmtx, …) are stored once; every face's directory just points its glyf record at the same byte. Only the small per-face metadata (name, OS/2, head) is duplicated. This is exactly what fontTools' TTCollection.save(shareTables=True) does, and it's why this whole course's hybrid font costs ~2 fonts of disk for 9 families.

2 · Variable fonts

A collection ships a few fixed faces. A variable font is the orthogonal answer: ship one set of master outlines plus deltas along axes, and let the renderer interpolate any face on demand. Common axis tags: wght (weight), wdth (width), opsz (optical size), slnt (slant), ital (italic).

tablerole
fvardeclares the axes and their ranges, plus named instances ("Bold", "Condensed")
gvarper-glyph point deltas for TrueType outlines (CFF2 is the CFF equivalent)
avaroptional non-linear remapping of an axis (so "50%" can mean what the designer wants)
STATstyle attributes — how named instances relate, for menus and fallback

Ask for weight 550 and the renderer interpolates between the masters at that location — a face that exists in no discrete file. The contrast with a collection is sharp:

approachhow it provides many styleshow many faces
TTC / OTCseveral fixed faces sharing static tablesa finite, designed set
variable fontone master set + deltas, interpolated at runtimeinfinite (a continuous space)

3 · Color fonts (briefly)

Plain outlines are monochrome. Several competing tables add color, and which one a platform honors varies:

  • COLR/CPAL — layered vector glyphs with a color palette (e.g. emoji on Windows). Scales cleanly.
  • sbix — Apple's embedded color bitmaps.
  • CBDT/CBLC — Google's embedded color bitmaps.
  • SVG — a table holding actual SVG documents per glyph.

4 · Web fonts: WOFF and WOFF2

This is the most common misconception, so state it plainly: WOFF and WOFF2 are not new font formats. They are compressed wrappers around the very same sfnt tables you already know. WOFF1 uses zlib/Deflate; WOFF2 uses Brotli and is roughly 30% smaller again. The browser decompresses the wrapper back into the identical table model and renders it exactly like a .ttf. Nothing about outlines, cmap, or metrics changes.

Subsetting is the other half of web delivery. Tools like pyftsubset drop the glyphs (and whole tables) a page doesn't use, then re-emit the font. For Latin this is a nicety; for CJK fonts it is vital — with tens of thousands of glyphs (this font has numGlyphs = 46830 from Module 4), the outlines dominate the file, and an unsubsetted CJK web font is a multi-megabyte download.

5 · Rasterization — closing the loop

Module 3 left you with a resolution-independent outline: contours of on/off-curve points in font units. A rasterizer — FreeType (Linux/Android), Core Graphics (Apple), DirectWrite (Windows) — scan-converts that outline into pixels at a target size. Along the way: hinting and grid-fitting nudge edges onto the pixel grid for crisp stems at small sizes, and antialiasing (greyscale, or subpixel/LCD using the R-G-B stripes) softens the staircase. That is the final step that turns the glyf table into the glyphs on this screen.

walkthrough

Dissecting LXGWHybrid.ttc

The collection build.py produces is the perfect specimen, because it is the whole course made physical. Its header begins with the collection magic "ttcf", and it carries 18 faces across 9 families in ~42 MB — almost exactly the size of the two source fonts (~25 MB WenKai + ~17 MB ZhenKai), not nine times a single font.

Why? Because the 18 faces only ever reference two sets of outlines. The nine Regular faces all reuse WenKai's glyf; the nine Bold faces all reuse ZhenKai's. The directory offsets prove it directly.

shareTables, proven by offset Across all 18 faces there are only two distinct glyf offsets:
  • glyf @ 368 — shared by faces [0,2,4,6,8,10,12,14,16] (the 9 Regular faces, WenKai outlines).
  • glyf @ 25848192 — shared by faces [1,3,5,7,9,11,13,15,17] (the 9 Bold faces, ZhenKai outlines).
Nine directories point at the very same outline bytes. The name tables, by contrast, have distinct offsets per face — that small per-family metadata is the only thing duplicated.

Walk three adjacent faces to feel the pattern:

facefamily (name id 1)subfamilyglyf offset
0LXGW WenKai ZhenKai Mono GBRegular368
1LXGW WenKai ZhenKai Mono GBBold25848192
2LXGW WenKaiRegular368

Face 2 belongs to a different family name than face 0, yet its outlines live at the same offset 368 — literally the same WenKai bytes. Nine family names, two sets of outlines, one file.

The whole course in one sentence build.py constructs 18 in-memory faces by retagging only metadata (Module 5: name/OS/2/head/post), leaving glyf/cmap/hmtx byte-identical (Module 1 table independence; Modules 3–4 outlines and metrics untouched). Then TTCollection.save(shareTables=True) collapses those identical heavy tables (Module 7) so 9 families cost ~2 fonts of disk.
See it for yourself xxd -l 16 LXGWHybrid.ttc shows the ttcf magic and face count; specimens/extract.py LXGWHybrid.ttc prints every face's family name and glyf offset so you can count the two distinct values yourself.

7.1concept

Where did the bytes go? The 18-face .ttc weighs ~42 MB — about the size of the two source fonts, not 9× a source font. Explain what makes that possible, and which tables are the exception that does get duplicated.

Show solution The 18 faces reference only two sets of physical tables: all 9 Regular faces share WenKai's glyf/cmap/hmtx (one offset), and all 9 Bold faces share ZhenKai's. shareTables=True stores each heavy table once and points many directories at it. The only per-face duplication is the small metadata that must differ — name, OS/2, head — which is tiny next to the 24.7 MB of outlines.
7.2concept

TTC vs variable font. Both let one file provide many styles. State the core mechanism of each in one line, and say when you'd reach for which.

Show solution TTC: several complete, fixed faces packed together, sharing identical static tables by offset. Reach for it when you have a small set of already-designed faces (or, as here, two unrelated fonts you want grouped) and want them to coexist cheaply. Variable font: one master set of outlines plus per-glyph deltas (gvar/CFF2) along axes declared in fvar; the renderer interpolates any location. Reach for it when you want a continuous design space (any weight 100–900, optical sizing) in the smallest file. TTC = a few static faces; variable = infinitely many interpolated faces.

7.4concept

What is a WOFF2? A teammate says "we should re-export the font in the WOFF2 outline format for the web." Is WOFF2 a different outline technology? If not, what is it actually, and why does the web prefer it?

Show solution No — WOFF2 is not an outline technology at all. It's a Brotli-compressed wrapper around the exact same sfnt tables (glyf/loca or CFF, cmap, hmtx, …). The browser decompresses it back to the identical table model and renders it like any TTF/OTF. The web prefers it purely for transfer size: Brotli plus font-specific preprocessing makes it ~30% smaller than WOFF1 and much smaller than raw TTF, with no quality loss. "Re-export in WOFF2" really means "compress the same font."
7.5concept

Why subset CJK especially? Subsetting matters for any web font, but it is almost mandatory for CJK ones. Using the glyph count you measured earlier (numGlyphs = 46830), explain why.

Show solution A Latin face has a few hundred glyphs; this CJK face has 46,830, and the glyf table alone is ~24.7 MB — the outlines dominate the file. A web page typically displays a few hundred to a few thousand distinct characters, so shipping all 46k glyphs wastes nearly the entire payload. pyftsubset keeps only the glyphs (and tables) actually used, turning a multi-megabyte download into something page-sized. Even with WOFF2's Brotli on top, the win comes mostly from not sending the unused glyphs. (Hence "dynamic subsetting" / unicode-range slicing for CJK in practice.)
7.6capstone

Capstone — read build.py with full understanding. Re-read lxgw-wenkai-zhenkai/build.py end to end. In your own words: (a) list every table it modifies and why; (b) name the one table-class it never touches and why that matters; (c) trace start to finish how "ask for bold in family X → get real ZhenKai strokes" actually works, citing Modules 1, 4, 5, and 7.

Show model solution

(a) Tables it modifies (all inside build_face, via set_name and direct field writes):

  • name — rewrites ids 1 (Family), 2 (Subfamily = "Regular"/"Bold"), 3 (Unique ID), 4 (Full name), 6 (PostScript name), and removes ids 16/17 (typographic family/subfamily). This is what makes the two files claim the same family with different subfamilies, so the OS groups them into one Regular+Bold family. Ids 3/6 are made unique per alias so CoreText doesn't dedup one file against another or against the originals (Module 5).
  • OS/2 — sets usWeightClass (400 Regular / 700 Bold), which is what "ask for bold" actually selects; flips fsSelection between the BOLD (0x20) and REGULAR (0x40) bits; and sets panose.bProportion (9 Monospaced / 3 Modern) for the Mono-named families.
  • head — sets macStyle's BOLD bit coherently with fsSelection (renderers cross-check the two; they must agree).
  • post — sets isFixedPitch so a terminal's font picker will accept the Mono families as monospaced — an advertised flag only, no widths change.

(b) The table-class it never touches: the outline/metric data — glyf/loca (outlines, Module 3), cmap (Module 2), and hmtx (advances, Module 4). It deliberately moves and redraws nothing. That matters for two reasons: (1) the CJK advance is already a full em in both fonts (一 = 1000/1000 em; build.py prints this equality as proof no re-metricizing is needed, so bold columns stay aligned — Module 4); and (2) leaving those tables byte-identical across every face built from the same source is precisely what lets TTCollection.save(shareTables=True) store them once (Module 7). Touching glyf would also break the on-disk 4-byte alignment and defeat sharing.

(c) "Ask for bold in family X → real ZhenKai strokes," start to finish:

  1. For each family name, build_face(wen, …, weight=400, bold=False) labels a copy of WenKai as that family's Regular, and build_face(zhen, …, weight=700, bold=True) labels a copy of ZhenKai as that family's Bold — same family name in name id 1, different subfamily/weight in name id 2 + OS/2.usWeightClass + fsSelection/macStyle (Module 5).
  2. All 18 faces are packed into one .ttc with shareTables=True; because the outline tables were never edited, the 9 Regular faces share WenKai's glyf at one offset and the 9 Bold faces share ZhenKai's at another — the two distinct offsets we proved (Modules 1 & 7).
  3. At runtime an app requests family "X" and asks for bold. CoreText / DirectWrite / fontconfig groups files by the baked-in family name and picks the member whose usWeightClass/style bits say Bold — that's the ZhenKai-derived face.
  4. The renderer follows that face's directory (Module 1), seeks to its glyf offset — which physically points at ZhenKai's outlines — and rasterizes them. The user sees genuine heavier ZhenKai strokes, not synthetic faux-bold, with columns still aligned because the advances were never altered (Module 4).

So the entire effect is achieved by editing four small metadata tables and exploiting container independence + offset sharing — never by drawing a single glyph.
