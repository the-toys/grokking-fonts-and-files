Module 03

Outlines

A glyph isn't a picture — it's a set of closed paths the renderer fills. In this font those paths are quadratic Bézier curves stored in glyf and located through loca. We'll go from "what a contour is" all the way down to hand-decoding a real glyph's bytes.

Exercises are interleaved — do each as you reach it. Programming ones have a collapsible ✓ Check your answer; concept ones have a solution.

concept

How a glyph is drawn

A glyph outline is one or more closed contours — loops of connected points. The renderer doesn't stroke the loops; it fills the region they enclose. When contours nest (the hole in an 'O', the counter in '永'), a winding rule decides what's inside: non-zero winding (the usual TrueType/PostScript rule) or even-odd. Reverse an inner contour's direction and, under non-zero winding, it punches a hole instead of adding ink.

Every coordinate is in font units on the em square. Here unitsPerEm = 1000, so (110, 383) means 110 units right, 383 up, on a 1000×1000 design grid. (How the em scales to pixels is all of Module 4; for now just read coordinates as positions on a 1000-unit grid.)

concept

Points, curves, and the two technologies

First, what a Bézier curve actually is

Before "on-curve / off-curve" can mean anything, you need the one idea underneath them. Start with a straight line from A to B. Drop a third point C off to one side and treat it as a magnet: bend the line so it bulges toward C, while still starting exactly at A, ending exactly at B, and never quite reaching C. That bent line is a quadratic Bézier curve.

          C ●   ← control point (the "magnet": the curve leans toward it, never touches)
         ·     ·
       ·         ·          ← the actual curve
  A ●·             ·● B
  on-curve          on-curve   (the curve DOES pass through A and B)

Two kinds of point fall out — and this is the entire vocabulary of glyph outlines:

  • On-curve points (A, B) — the path passes through them. Also called anchors: the outline is pinned to them. (So "point 0 = anchor" just means point 0 is on-curve.)
  • Off-curve points (C) — the path only leans toward them. Also called control points: C far out → big bulge; C on the A–B line → no bulge (straight).

Quadratic vs cubic is just how many magnets per segment: a quadratic has one control point (a simple arc — TrueType); a cubic has two, one near each end (enough for an S-curve in one segment — CFF/PostScript). A quadratic can't make an S in one segment, so TrueType chains several — which is why TrueType glyphs carry more points than CFF for the same shape.

A contour is then just a loop of anchors with control points between: two adjacent anchors (1 1) = a straight edge; anchor–control–anchor (1 0 1) = one curved segment; two controls in a row (0 0) = TrueType invents an anchor at their midpoint (next section).

The two technologies

Which curve math a font uses is announced by its sfnt magic from Module 1:

  • TrueType (this font — magic 00 01 00 00): quadratic Béziers. Outlines in glyf, located via loca. Each point carries a flag: on-curve (an anchor the path passes through) or off-curve (a quadratic control point that bends the curve toward it without being touched).
  • CFF / CFF2 (PostScript-flavoured OTF — magic OTTO): cubic Béziers, stored in a CFF  table (trailing space — still a 4-byte tag). No glyf/loca.
Quadratic vs cubic, in one line A quadratic segment has one control point; a cubic has two. Cubics express an S-curve in one segment (CFF often needs fewer points); TrueType compensates with cheaper math and the "implied on-curve point" trick below. Same shapes, two encodings.

The implied on-curve point (TrueType)

The shortcut that trips up everyone: when two off-curve points sit next to each other, an on-curve point is implied at their exact midpoint. The font stores fewer anchors; the renderer reconstructs them. So a run of off-curve points isn't one impossible mega-curve — it's a chain of quadratic segments joined at implied midpoints.

concept

loca, composites, hinting

loca is to glyf what the table directory was to the file: an index of byte offsets. Entry i and i+1 give the start and end of glyph i's data inside glyf — so loca holds numGlyphs + 1 entries (you need the offset after the last glyph to bound it). If loca[i] == loca[i+1], glyph i has zero bytes — an empty glyph like space. head.indexToLocFormat picks the entry size: 0 = 16-bit short offsets (stored as offset ÷ 2), 1 = 32-bit long offsets stored directly.

Composite glyphs

A glyph need not draw its own outline. A composite is built from references to other glyphs (components), each placed with an offset and optionally a scale/2×2 transform. 'Ä' = "draw 'A', then the dieresis shifted up". Huge for CJK (thousands of characters share radicals) and accented Latin (base + combining mark). It saves space (one outline, many refs) and enforces visual consistency (every accent is literally the same shape).

Hinting, briefly At tiny sizes a stem can blur or vanish. The prep/fpgm programs (TrueType bytecode) plus gasp control grid-fitting — nudging points onto pixel boundaries. WenKai's prep is just 7 bytes (Module 1), and modern antialiased rendering leans on hinting far less than 1990s 1-bit displays did. Know it exists; don't overstate it.
3.4concept

Why composites. Why are they especially valuable for CJK and accented Latin? Name the two things they save.

Show solution Space and visual consistency: store a shared radical / base / mark once and reference it with a placement offset, instead of duplicating thousands of outlines; and because every reference points at the same outline, the shape is identical everywhere.
walkthrough

Reading real outlines from WenKai

This font's head.indexToLocFormat = 1, so loca uses 32-bit long offsets — direct byte offsets into glyf (which is 24.7 MB, so 16-bit couldn't reach anyway).

'一' (U+4E00) — an off-curve run

"One", a single horizontal stroke: numberOfContours=1, 21 points, endPtsOfContours=[20]. First six points:

point:   0        1        2        3        4        5
coord:  (110,383)(837,420)(861,422)(881,430)(893,430)(921,413)
onCurve:  1        1        0        0        0        0
          └anchor  └anchor  └────── off-curve run ──────┘
  • 0 → 1: two on-curve anchors in a row ⇒ a straight line (no control point between) — the long top edge.
  • 2–5: a run of four control points. Implied midpoints between each adjacent pair turn it into chained quadratic segments rounding the stroke's right end. The path never touches 2–5; they only pull it.

'十' (U+5341) and the composite dieresis

"Ten" — a cross: 1 contour, 46 points, flags beginning 1 1 0 1 0 1 (an alternating anchor/control rhythm — straighter strokes with single curves at the corners). And the spacing dieresis (¨) is a composite: it has no outline of its own — its whole record is "draw the dieresiscomb glyph, shifted to (500, 0)". One outline, reused.

binary deep-dive

First: navigate via loca to a glyph's bytes

To hand-parse a glyph you must first find it — and a 21-point glyph like '一' is a lot to decode by hand, so we'll use the smallest real glyph that still shows everything: '─' (U+2500), a single 4-point rectangle (a horizontal box-drawing bar). The navigation chain, using the table offsets from Module 1 (glyf @ 592352, loca @ 405028):

name         = getBestCmap()[0x2500]        = "boxdrawingslighthorizontal"
gid          = getGlyphID(name)             = 869
loca[869]    = 153477   (long/32-bit offset into glyf)
loca[870]    = 153501
glyph length = 153501 − 153477             = 24 bytes
file offset  = glyf_base + loca[869]
             = 592352 + 153477             = 745829

Note glyph names vary — '─' is not "u2500"; resolve the name through the cmap (getBestCmap()[0x2500]) rather than assuming it. '一' happens to be "u4E00", but that pattern isn't guaranteed.

The 24 raw bytes at that offset:

00 01 00 00 01 68 03 e8 01 90 00 03 00 00 11 35 21 15 03 e8 01 68 28 28

The compressed bit: flags & coordinates

After the header come the flags (one per point, but run-length compressible) and then the coordinates. The flag byte is a bitfield:

bitmasknamemeaning
00x01ON_CURVE1 = on-curve anchor, 0 = off-curve control
10x02X_SHORTx delta is 1 byte (else 0 or 2)
20x04Y_SHORTy delta is 1 byte
30x08REPEATnext byte = how many extra points repeat this flag
40x10X_SAME / +dual: if X_SHORT → sign (1=+, 0=−); else → x delta is 0 (same as prev)
50x20Y_SAME / +dual: if Y_SHORT → sign; else → y delta is 0

Our four flag bytes 11 35 21 15 (no REPEAT bit set, so one byte per point):

ptflagbits seton-curve?
00x11ON_CURVE, X_SAME/+yes
10x35ON_CURVE, Y_SHORT, X_SAME/+, Y_SAME/+yes
20x21ON_CURVE, Y_SAME/+yes
30x15ON_CURVE, Y_SHORT, X_SAME/+yes
Three rules that make coordinates tiny (1) Deltas: each coordinate is stored relative to the previous point (the first relative to origin 0). (2) All-X-then-all-Y: every x delta is stored first, then every y delta — not interleaved. (3) Variable width per flag: if the SHORT bit is set → 1 byte (sign from the SAME/+ bit); else if the SAME/+ bit is set → 0 bytes (delta = 0); else → a 2-byte signed int16. That's why the same/zero deltas in our glyph cost nothing.

Decoding the trailing 03 e8 01 68 28 28 — x deltas first (4 of them), then y deltas:

X (flags say: pt0 SAME→0, pt1 SAME→0, pt2 int16, pt3 SAME→0)
   pt2 int16 = 03 e8 = +1000      ⇒ x = 0, 0, 1000, 1000     (consumes "03 e8")
Y (pt0 int16, pt1 SHORT +, pt2 SAME→0, pt3 SHORT −)
   pt0 int16 = 01 68 = +360
   pt1 short = 28 = +40           (Y_SHORT set, Y_SAME/+ set ⇒ positive)
   pt3 short = 28 = −40           (Y_SHORT set, Y_SAME/+ clear ⇒ negative)
                                   ⇒ y = 360, 400, 400, 360  (consumes "01 68 28 28")
point(x, y)on-curve
0(0, 360)yes
1(0, 400)yes
2(1000, 400)yes
3(1000, 360)yes

Four on-curve corners ⇒ a filled rectangle from (0,360) to (1000,400): a full-width bar 40 units tall at mid-height — exactly '─'. All 24 bytes accounted for, and it matches the bounding box in the header.

The two traps Coordinates are all-X-then-all-Y (read every x delta before any y), and bits 4/5 are context-dependent — they mean "sign" when the matching SHORT bit is set, but "delta is zero / same as previous" when it isn't. Treat them as a fixed meaning and your coordinates drift.

A composite glyph in bytes: dieresis

The spacing dieresis has numberOfContours = −1 (the composite signal), then one component record:

No MORE_COMPONENTS bit ⇒ it's the only component. So the entire glyph is "place dieresiscomb at (500, 0)" — the composite mechanism in its smallest form.

tooling & introspection

Exploring outlines yourself

Same recipe as before — type / dir / help. A glyph object (font['glyf']['u4E00']) exposes:

memberwhat it gives
.numberOfContours≥0 simple, −1 composite
.coordinates / .flagsthe points and their parallel flag bytes
.endPtsOfContourslast point index of each contour
.isComposite()True for component-based glyphs
.getComponentNames(glyf)component glyph names (composites)
.draw(pen) / .drawPoints(pen)replay the outline into a pen (see below)
.programthe hinting bytecode, if any
.compile(glyf)serialize back to the raw bytes we hand-parsed

The most useful idea here is the pen protocol: outlines are replayed as drawing commands (moveTo, lineTo, qCurveTo, closePath) into a "pen" object. RecordingPen captures them as a list; BoundsPen measures; SVGPathPen emits SVG. It's how you turn raw points into something usable without re-deriving the implied-midpoint math by hand:

from fontTools.pens.recordingPen import RecordingPen
pen = RecordingPen()
name = font.getBestCmap()[0x2500]        # "boxdrawingslighthorizontal" — not "u2500"!
font.getGlyphSet()[name].draw(pen)
pen.value   # [('moveTo', ...), ('lineTo', ...), ('closePath', ())]
