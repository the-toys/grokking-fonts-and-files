Module 03

Outlines

A glyph is a set of closed paths the renderer fills. We'll build up in the natural order: what stores the shapes (glyf), how you find one glyph's data (loca), how the two work together, what the resulting numbers mean, and finally how to decode the raw bytes by hand.

concept · 1

glyf — where the shapes live

The glyf table is one big blob holding every glyph's outline, one variable-length record per glyph, packed back to back. A record describes the glyph as contours — closed loops of points — plus a bounding box. The renderer fills the region those loops enclose; it doesn't just stroke them.

Which kind of curve those points describe is set by the font's technology, announced by the sfnt magic from Module 1:

  • TrueType (this font, magic 00 01 00 00): outlines in glyf, made of quadratic curves. (We unpack what that means in step 4.)
  • CFF / CFF2 (magic OTTO): outlines in a CFF  table instead, made of cubic curves. No glyf at all.

For now just hold this: glyf = the outline data; one packed record per glyph. The next question is how you find one glyph inside that 24.7 MB blob.

concept · 2

loca — finding one glyph's bytes

Because glyf records are variable length (a comma has a few points, a CJK character hundreds) and packed with no separators, you can't jump to glyph i without a map. That map is loca — an array of byte offsets into glyf, one entry per glyph plus one — so numGlyphs + 1 entries, where numGlyphs is the font's total glyph count (stored in the maxp table from Module 1; 46830 here, as Module 2 used):

loca:   off0      off1          off2       off3   …   offN+1   (N+1 entries, N = numGlyphs)
        │         │             │          │
        ▼         ▼             ▼          ▼
glyf:   ┌─────────┬─────────────┬──────────┬─ …              (variable-length records, packed)
        │ glyph 0 │   glyph 1   │ glyph 2  │
        └─────────┴─────────────┴──────────┴─ …

        each loca[i] marks the START of glyph i, so   glyph i's bytes = glyf[ loca[i] : loca[i+1] ]

Three consequences worth saying out loud:

  • Why N+1 entries: glyph i runs from loca[i] to loca[i+1], so the very last glyph needs an offset after it to mark its end.
  • Empty glyphs: if loca[i] == loca[i+1] the glyph has zero bytes — it draws nothing (e.g. the space character). Perfectly normal.
  • Two storage formats: the entry size is picked by one field, written head.indexToLocFormat ("the indexToLocFormat field of the head table"). head is the OpenType font header table — a named table reached through the directory, listed in Module 1, not the 12-byte file header at offset 0. 0 = 16-bit "short" offsets stored as the real offset ÷ 2 (saves space, but only reaches 128 KB of glyf); 1 = 32-bit "long" offsets stored directly. This font uses 1 (its glyf is 24.7 MB — far past what short offsets could address).

walkthrough · 3

glyf + loca together (and our first real coordinates)

Put them together to fetch one glyph. Take '一' (U+4E00). First codepoint → glyph id (via the cmap from Module 2), then glyph id → byte range (via loca):

name  = getBestCmap()[0x4E00]   = "u4E00"
gid   = getGlyphID(name)        = 3171
loca[3171]   = 931382      ┐  glyph 3171's bytes are
loca[3172]   = 931454      ┘  glyf[931382 : 931454]  →  72 bytes long

Those 72 bytes are the outline. fontTools decodes them into a tidy object so you can read the shape without touching bytes yet:

from fontTools.ttLib import TTFont
font = TTFont(path)
g = font["glyf"]["u4E00"]
g.numberOfContours      # 1   (>0 simple glyph, -1 composite)
g.coordinates           # 21 ABSOLUTE (x,y) points: [(110,383), (837,420), …]
g.flags                 # parallel bytes; bit0 (& 1) = on-curve
g.endPtsOfContours      # [20]  last point index of each contour

So glyph '一' is 1 contour, 21 points, and its first few coordinates are (110,383) (837,420) (861,422) (881,430) …. Those raw numbers are what step 4 makes sense of. (And yes — this is exactly the navigation the table directory does for the whole file, one level down: an index of offsets pointing into a packed blob.)

concept · 4

What each coordinate means

Now the payoff. We have a list of points like (110, 383) and a flag per point. Two things to decode: where a point is, and what role it plays.

Where: font units, measured from the baseline

Coordinates are in font units: x → right from the origin (the left edge), y → up from the baseline — the line text sits on, at y = 0. unitsPerEm = 1000 is the scale (1 em = 1000 units), so (110, 383) means 110 units right, 383 above the baseline.

Coordinates are signed — and not boxed into [0, 1000] The baseline is not the bottom of the glyph; it's the writing line. Anything below it has negative y — Latin descenders ('g' → −175, 'p', 'y', 'j') and CJK strokes that dip down ('十''s vertical reaches y = −86). Ink can also sit left of the origin (negative x) or extend past the em. The 1000 is a ruler, not a bounding box — glyphs aren't confined to it. (So when g.coordinates shows (537, −59), that's just "x 537, 59 units below the baseline".)

How units scale to pixels is all of Module 4; for now, read each point as a position on this baseline-relative ruler.

Role: on-curve vs off-curve — and what a Bézier curve is

Each point's flag bit 0 marks it on-curve or off-curve. To see why that matters, here's the one idea underneath: start with a straight line from A to B, then drop a third point C off to the side and treat it as a magnet — bend the line so it bulges toward C, while still starting at A, ending at B, and never reaching C. That bent line is a quadratic Bézier curve.

          C ●   ← off-curve "control" (the magnet: curve leans toward it, never touches)
         ·     ·
       ·         ·          ← the actual curve
  A ●·             ·● B
  on-curve          on-curve   (the curve DOES pass through A and B)
  • On-curve points (A, B) — the path passes through them; also called anchors (the outline is pinned to them).
  • Off-curve points (C) — the path only leans toward them; also called control points. C far out → big bulge; C on the A–B line → straight.

Quadratic vs cubic = how many magnets per segment: quadratic has one control point (TrueType); cubic has two, one near each end (CFF), enough for an S-curve in a single segment. A quadratic can't make an S in one go, so TrueType chains several — why TrueType glyphs carry more points than CFF for the same shape.

Reading a flag string

A contour is a loop of these points. The on-curve flags tell you the segments:

  • 1 1 — two anchors in a row, no magnet between ⇒ a straight edge.
  • 1 0 1 — anchor, control, anchor ⇒ one curved segment.
  • … 0 0 … — two off-curve points in a row ⇒ TrueType implies an on-curve point at their midpoint, so a run of controls becomes a smooth chain of arcs.

So '一''s first six flags 1 1 0 0 0 0 read as: anchor, anchor (the straight top edge), then a run of four controls (the rounded right end), with implied midpoints between them. Here it is literally, on the em square:

Contours, holes, and winding

A glyph is one or more closed loops (contours), and the renderer fills their inside. The only wrinkle: when loops nest — the bowl of an 'O', the frame of '口' — it has to decide whether the enclosed region is more ink or a hole. One rule decides every case.

The rule

From the point you're testing, shoot a ray off to infinity and count the contour edges it crosses — but with a sign: each crossing is +1 or −1 by which way the edge crosses (one direction is +1, the opposite −1 — which is which is just a fixed convention). The point is inside when the signed total isn't zero, outside when it's zero. That's the non-zero winding rule. (A simpler cousin, the even-odd rule, drops the sign and fills on an odd count; TrueType uses non-zero.)

Why nesting makes a hole

The whole trick is one fact: an inner loop is drawn the opposite direction from the outer one. Trace a ray outward from each kind of point:

  • In the solid ring (between outer and inner): the ray crosses only the outer edge — one crossing, so the total can't be zero ⇒ filled. (Its sign is whatever the convention says — here −1, because the outer winds clockwise — but only nonzero-vs-zero matters, never the sign itself.)
  • In the enclosed region: it crosses the inner edge and the outer edge, and because they wind opposite ways those two crossings take opposite signs and cancel (+1 and −1 → 0) ⇒ empty — a hole.
  • Outside everything: no crossings ⇒ 0 ⇒ empty.
It's the opposite winding, not the nesting Draw that inner loop the same way as the outer and the two crossings take the same sign — they add instead of cancelling (to ±2, never zero) — so the "hole" fills in solid. Direction is what carves a hole; one loop merely sitting inside another isn't enough.

So several contours do not mean a hole: '永' is three strokes all wound the same way — three separate filled blobs, zero holes. And a Latin 'O' is exactly the '口' picture: one outer loop, one opposite-wound inner loop. The real cases:

Why the ray's direction doesn't matter

That recipe should make you suspicious: if I shoot the ray straight up instead of to the right, do I get the same answer? Yes — always, and seeing why is the heart of the whole rule, so let's be careful.

Forget the ray for a moment. Stand at the test point P and watch a pen trace the entire contour once around. Track only the direction from P to the pen — an angle that swings as the pen moves. When the pen returns to its start, that angle returns too, but along the way it may have wound around P some whole number of full turns. That integer — counter-clockwise turns minus clockwise turns — is the winding number of P. Read that definition again: it mentions no ray at all. It depends only on P and the curve.

Ray-casting is merely a way to count those turns. Each time the contour crosses your ray, the pen's direction is sweeping past the ray's one fixed angle — once in the +turn sense (count +1) or the −turn sense (−1). Over the full loop, the signed crossings add up to exactly the net number of turns — the winding number. Since that number contains no ray, every ray, in every direction, must report the same total. The ray is just a tick-mark for counting turns; where you put the tick can't change how many turns the pen actually made.

The same fact, seen as cancellation Here it is without the angle argument. Rotate your ray slowly from one direction toward another. The signed crossing count can only change at the instant the ray grazes the curve tangentially — and there, two crossings are always born (or destroyed) together, with opposite signs: the ray dips across the edge and back out, a +1 and a −1 that cancel. So the signed sum never moves as the ray turns ⇒ all directions agree. (The even-odd rule survives for the same reason: crossings still appear and vanish in pairs, so the parity of the count can't change either.)

One honest caveat: choose a generic ray — one that doesn't pass exactly through a listed point or run along an edge. Those degenerate directions are only finitely many; real rasterizers sidestep them with a fixed tie-break (e.g. "count an edge only at its lower endpoint") so a boundary pixel is never double-counted or missed.

How a renderer actually fills it

The per-point ray is the definition of inside, not the algorithm — no rasterizer shoots a separate ray per pixel. It works a row at a time (scanline conversion):

  1. Take a horizontal scanline across a pixel row (at the row's centre).
  2. Find where contour edges cross it — just the x-positions, each tagged +1/−1 by direction.
  3. Sort those x's and sweep left→right keeping a running winding sum; it changes only at a crossing, so fill each span where the sum is nonzero.

One scanline answers a whole row at once — the "ray" is the scanline, reused for every pixel on it, not recast per point.

The awkward hits (the scanline grazing a vertex, or lying along a horizontal edge) aren't fixed by rotating the ray — they're settled with a fixed tie-break: treat each edge as half-open in y (count it at its lower endpoint, not its upper) and ignore exactly-horizontal edges. That counts every shared vertex exactly once, so neighbouring spans share a boundary cleanly — no double-draw, no one-pixel crack. (It's the same "top-left rule" GPUs use for triangles, and it's only sound because any ray direction agrees — which we just proved.)

Two layers ride on top. Anti-aliasing: rather than a flat in/out, good rasterizers compute each pixel's fractional coverage (how much of it the outline covers) by sub-sampling or by accumulating exact signed area per pixel cell (FreeType's "smooth" rasterizer) — same winding logic, measuring area instead of yes/no. Hinting: at small sizes the prep/fpgm/gasp machinery grid-fits the outline onto pixel boundaries before the fill runs, so stems stay crisp.

The fine print: direction, role, and orientation

Direction vs. role. A contour's direction (clockwise / counter-clockwise, judged with y up) is exactly the sign of its signed (shoelace) area — checkable per contour. But whether it's the outer boundary, a hole, or a solid island inside a hole is a containment fact the sign can't give you. The convention this font follows (TrueType; PostScript/CFF is reversed): the outermost contour is clockwise, and each level of nesting flips the winding (CW, CCW, CW, …), so solid/hollow alternate inward. "Outer CW, hole CCW" is only the two-level case — '回' nests four deep, so its third contour is clockwise yet an inner ring, not the outer.

When is "clockwise" even defined? Walk a loop that never crosses itself and your facing direction turns exactly one full circle — +360° (CCW) or −360° (CW); that sign is the orientation, and the signed area computes it. It stops being defined for a self-intersecting path (a figure-eight turns each way on its two lobes; a symmetric one even has signed area exactly 0) or a degenerate zero-area path. Neither occurs in clean glyphs — and the rasterizer never needs a global label anyway, since the winding number is defined point-by-point from local crossings.

binary deep-dive · 5

Hand-parsing a glyph's bytes

We've used the library; now decode the raw bytes ourselves. '一' has 21 points (a lot to do by hand), so we'll use the smallest real glyph that still shows everything: '─' (U+2500), a single 4-point rectangle. First navigate to its bytes (table offsets from Module 1: glyf @ 592352, loca @ 405028):

name         = getBestCmap()[0x2500]   = "boxdrawingslighthorizontal"   (not "u2500"!)
gid          = getGlyphID(name)        = 869
loca[869]    = 153477                   ┐ length = 153501 − 153477 = 24 bytes
loca[870]    = 153501                   ┘ file offset = 592352 + 153477 = 745829

Note glyph names vary — resolve via the cmap (getBestCmap()[0x2500]) rather than assuming "u2500". '一' happens to be "u4E00", but that pattern isn't guaranteed.

How to actually grab the bytes from loca loca is an array of N+1 offsets (one extra sentinel at the end). Glyph gid's data is the half-open range [loca[gid], loca[gid+1]) — you read the glyph's own offset and the next one, and their difference is the length. That trailing sentinel is exactly why the array has one more entry than there are glyphs: even the last glyph needs a gid+1 to bound it. Three things to get right:

(1) loca offsets are relative to the glyf table, not the file. Add the glyf table's own offset (from the Module 1 directory) to seek in the raw file: file_off = glyf_offset + loca[gid].
(2) Short vs long loca format (head.indexToLocFormat): 0 = short, each offset is a uint16 storing offset ÷ 2 (multiply by 2, glyphs are 2-byte aligned); 1 = long, each is a uint32 byte offset used as-is. This font is long. fontTools' font['loca'].locations hands you already-decoded byte offsets either way — the ÷2 only bites when parsing loca's bytes by hand.
(3) Zero length ⇒ no outline. If loca[gid] == loca[gid+1] the glyph has no glyf data at all — a blank glyph like space. Read nothing; don't parse a header. 
import struct
glyf_off   = font.reader.tables['glyf'].offset    # 592352, from the sfnt directory
loca       = font['loca'].locations           # decoded offsets, len = numGlyphs + 1
gid        = 869
start, end = loca[gid], loca[gid+1]            # 153477, 153501
length     = end - start                     # 24   (0 ⇒ blank glyph, stop here)
with open(path, "rb") as fh:
    fh.seek(glyf_off + start)                # 745829
    raw = fh.read(length)
# raw = 00 01 00 00 01 68 03 e8 01 90 00 03 00 00 11 35 21 15 03 e8 01 68 28 28

'─' is the easy case — one contour, no hinting. The general simple-glyph layout has two variable-length fields hiding in those two rows:

The two fields that aren't fixed-width (1) endPtsOfContours is n uint16s, so 2·n bytes — '─' has one contour (2 bytes), but a 5-contour glyph like this font's .notdef carries 00 21 00 46 00 93 00 c9 00 d1 = (33, 70, 147, 201, 209), ten bytes. Each value is the index of that contour's last point (cumulative, not a count): contour 0 is points 0–33, contour 1 is 34–70, and so on. Crucially, the last entry + 1 is the total point count (209 + 1 = 210) — that's the number you need before you can read the flags, since there's one flag per point.

(2) instructionLength bytes of hinting follow it. If instructionLength = m is nonzero, exactly m bytes of TrueType hinting bytecode sit between instructionLength and the flags. They nudge points onto the pixel grid at specific render sizes — they do not change the stored outline, so when hand-parsing you just skip m bytes to reach the flags. (Every glyph in this font has instructionLength = 0 — common for large CJK fonts that lean on the renderer's auto-hinter rather than hand-hinting tens of thousands of glyphs — which is why '─' shows 00 00 and the flags start immediately.)

Flags & the delta-coded coordinates

After the header come the flags (one per point, run-length compressible), then the coordinates. The flag byte is a bitfield. The x delta is controlled by a pair of bits (1 and 4); the y delta by the mirror pair (2 and 5):

bitmasknamemeaning
00x01ON_CURVE1 = on-curve anchor, 0 = off-curve control
10x02X_SHORT1 ⇒ x delta occupies 1 byte; 0 ⇒ it occupies 0 or 2 bytes
20x04Y_SHORTsame, for y
30x08REPEATnext byte = how many extra points repeat this flag
40x10X_SAME_OR_SIGNmeaning depends on X_SHORT — see the 2×2 below
50x20Y_SAME_OR_SIGNsame, for y
One byte per point, not 6 bits Each flag is a whole byte. Only bits 0–5 are defined above (bits 6–7 are reserved), but the parser still reads one full byte per point — flags are byte-aligned and never bit-packed across point boundaries, so you never "peel 6 bits then straddle into the next byte." (That byte-alignment is exactly what lets REPEAT work: it duplicates an entire identical flag byte.)

Within a byte, "bit N" is the bit worth 2N, counted from the least-significant end: bit 0 = 0x01 is the LSB, bit 5 = 0x20. You test one with flag & mask. Peeling 0x35 = 0011 0101 from the LSB up: bits 0, 2, 4, 5 are set (so ON_CURVE, Y_SHORT, X_SAME_OR_SIGN, Y_SAME_OR_SIGN — note 0x35 & 0x02 = 0, X_SHORT is clear).

Bit 4 is the one that trips everyone up: it has two unrelated meanings, and bit 1 (X_SHORT) decides which. The spec name spells out both halves — X_IS_SAME_OR_POSITIVE_X_SHORT_VECTOR: when the delta is short, bit 4 is its sign; when it isn't, bit 4 means "same as the previous point" (a zero delta). So the two x-bits together select exactly one of four cases:

bit 1 (X_SHORT)bit 4bytes readx delta
111+ (that byte, 0–255)
101− (that byte, 0–255)
0100 — same as previous point
002signed int16

(The short form is an unsigned byte with its sign supplied by bit 4 — that's how 1 byte still reaches ±255. The y delta works identically off bits 2 and 5.)

Our four flag bytes 11 35 21 15 (no REPEAT bit, so one per point): all four have bit 0 set ⇒ all on-curve. Two more rules govern how the coordinate bytes that follow are read:

Two rules for reading the coordinate stream (1) Deltas (a running sum): each stored value is a step from the previous point, so a point's absolute coordinate is the running total — start at 0 (the first step is measured from the origin) and add each delta in turn; x and y accumulate independently. (2) All-X-then-all-Y: every x delta first, then every y delta — not interleaved. How wide each delta is (0, 1, or 2 bytes) and its sign come straight from the 2×2 table above.

Now decode the trailing 6 bytes 03 e8 01 68 28 28. Work in two passes — read every x delta, then every y delta — and walk a cursor through the bytes, consuming exactly the width each flag dictates. Step 1: from the four flags, read off each point's x and y width (using the 2×2 table):

Four on-curve corners ⇒ a filled rectangle (0,360)→(1000,400): a full-width bar 40 units tall at mid-height — exactly '─', matching the header's bounding box. All 24 bytes accounted for. (Note the −40 was a delta, not a coordinate — here the running sum stays ≥ 0. But it needn't: a glyph with a descender carries the y total below the baseline into negatives — e.g. 'g' reaches y = −175 — which is exactly why every coordinate field is a signed int16.)

Bytes vs. g.coordinates — don't confuse the layers This delta / running-sum encoding lives only in the raw glyf bytes. fontTools' g.coordinates (step 3) are already the absolute points — it runs this accumulation the instant it loads the glyph. So '十''s g.coordinates = [(536,386), (537,−59), …] are positions you can plot directly (and they look right); treat those as deltas and accumulate and you'd get (536,386), (1073,327), … — x past the 1000 em, nonsense. You only do the running sum by hand when parsing the raw bytes yourself, as in this section.
The two traps Coordinates are all-X-then-all-Y (read every x delta before any y), and bits 4/5 are context-dependent — "sign" when the matching SHORT bit is set, "delta is zero" when it isn't. Treat them as one fixed meaning and your coordinates drift.

A composite in bytes: dieresis

A composite glyph draws no outline of its own — it places other glyphs, each as a component record: a 16-bit flags word, the referenced glyph's index, then placement args (and, optionally, a transform). The spacing dieresis has numberOfContours = −1, then one such record:

ff ff 00 6c 02 50 01 88 02 bd 00 07 a4 67 01 f4 00 00
bytesfieldvalue
ff ffnumberOfContours−1 ⇒ composite

The flags word drives everything after it. Just as the simple glyph's coordinate bits set each delta's width, these 16 bits make the component record variable-length — they decide how wide the args are, what the args mean, and whether a transform and further components follow:

bitmasknameeffect on the record
00x0001ARG_1_AND_2_ARE_WORDSset ⇒ the two args are int16 (2 bytes each); clear ⇒ int8 (1 byte each)
10x0002ARGS_ARE_XY_VALUESset ⇒ args are an (x, y) offset; clear ⇒ args are a point-match pair (see below)
20x0004ROUND_XY_TO_GRIDround the placement offset to the integer grid
30x0008WE_HAVE_A_SCALEone F2Dot14 scale follows the args (applied to x and y)
50x0020MORE_COMPONENTSanother component record follows; clear ⇒ this is the last
60x0040WE_HAVE_AN_X_AND_Y_SCALEtwo F2Dot14 scales (sx, sy) follow
70x0080WE_HAVE_A_TWO_BY_TWOa full 2×2 matrix (4 × F2Dot14) follows
90x0200USE_MY_METRICSthe composite inherits this component's advance width / lsb

Ours is 0x0007 = bits 0+1+2 ⇒ ARG_1_AND_2_ARE_WORDS | ARGS_ARE_XY_VALUES | ROUND_XY_TO_GRID. So the args are int16 (that's why 01 f4 00 00 is 4 bytes, not 2), they're an xy offset, and that offset is rounded to the grid. No scale or 2×2 bit is set ⇒ the transform is the identity — pure translation, no scaling or rotation.

glyphIndex a4 67 = 0xa467 = 42087 is a raw glyph ID into this same font; index 42087's outline is the combining diaeresis (the readable name "dieresiscomb" comes from post/cmap, not from these bytes). It's resolved through loca/glyf like any glyph — so the reference is recursive: a component can itself be a composite, nesting until it bottoms out in real outlines.

arg1, arg2 01 f4 00 00 = (0x01f4, 0x0000) = (500, 0). Because ARGS_ARE_XY_VALUES is set, these are a placement offset added to every point of the component before it's drawn (shift the diaeresis mark 500 units right, 0 up). Were that bit clear, the very same two values would instead be a point-match pair — "weld point arg1 of the glyph-so-far onto point arg2 of this component" — the alternative anchoring mechanism some fonts use to snap a mark onto a base.

No MORE_COMPONENTS bit (0x0020) ⇒ this is the only component. The whole glyph is "place glyph 42087 at offset (500, 0), identity transform." That's why composites matter for CJK (share a radical) and accented Latin (base + mark): store one outline, reference it many times — saving space and keeping every copy visually identical.

More than one component

A composite usually places several glyphs. There's no count field — the records are simply concatenated, and each one's MORE_COMPONENTS bit (0x0020) says "another record follows." The parser loops: read a record, and if that bit was set, read the next; the last record leaves it clear. (Both fonts in this tutorial happen to use only single-component composites, so the bytes below are a constructed example — an accented 'ä' built as base 'a' + the diaeresis mark — but the layout is exactly what a real one looks like:)

ff ff <8 bbox bytes>  00 27 00 79 00 00 00 00  00 07 a4 67 01 f4 02 58

Each record is independently flagged, so components can mix widths and transforms — one with int8 args and identity, the next with int16 args and a scale. After the final record, one more thing can appear: if any component set WE_HAVE_INSTRUCTIONS (0x0100), a uint16 instruction length and that many hinting bytes close out the glyph. Otherwise the glyph ends the moment the MORE_COMPONENTS chain stops.

tooling & introspection

Exploring outlines yourself

Same type/dir/help recipe. A glyph object exposes .numberOfContours, .coordinates, .flags, .endPtsOfContours, .isComposite(), .getComponentNames(glyf), .draw(pen)/.drawPoints(pen), .program (hinting), and .compile(glyf) (back to the raw bytes).

The big idea here is the pen protocol: outlines are replayed as drawing commands (moveTo, lineTo, qCurveTo, closePath) into a "pen". RecordingPen captures them; BoundsPen measures; SVGPathPen emits SVG. It turns raw points into something usable without redoing the implied-midpoint math:

from fontTools.pens.recordingPen import RecordingPen
pen = RecordingPen()
name = font.getBestCmap()[0x2500]        # "boxdrawingslighthorizontal"
font.getGlyphSet()[name].draw(pen)
pen.value   # [('moveTo', …), ('lineTo', …), ('closePath', ())]
