两道字体 CTF 挑战

屏幕上你看到的文字，只是一条流水线的末端：码位经 cmap 变成字形 ID （模块 2），字形 ID 经 glyf 变成轮廓（模块 3），整串还要经整形（模块 6）改写一遍， 然后才落下第一个像素。这条链上的每一站，都能藏东西——这正是“字体”成为 CTF 常客的原因。下面两道题各是一种经典套路，每道都附一只真实的、极小的字体供你下载， 用本课程已经教过的工具就能破。

两只字体都由 specimens/ctf/build_ctf.py 从样本字体子集化重建， 脚本会自我校验（写文件前先把 flag 复原出来）。这里没有一处是摆设——flag 是真真切切藏在字节里的。 flag 形如 flag{…}。

两道题背后是同一个念头 字体渲染出来的东西，和它的字节字面上说的东西，本是两码事，中间靠一张你读得懂的 查找表相连。题目就把 flag 藏在这道缝里：第一题藏进 cmap（字符→字形的查找表）， 第二题藏进 GSUB（字形→字形的改写）。破题之道，就是去读那张查找表，而不是 轻信渲染结果。
挑战 · cmap

1 · 渲染出来的文字在撒谎

某个爬虫从页面上扒下一段文字，拿到的却是一串“豆腐块”——私用区 （U+E000、U+E001……）里的码位，在任何正常字体里都显示成空方框。可那个页面 自带了一只字体，用它来渲染时，同样这串码位却拼出了一个 flag。字体在你手上，把 flag 找出来。

cmap_swap.ttf leaked.txt

leaked.txt 就是爬虫抓到的那串诱饵：码位 U+E000 … U+E017，按序排列。 用 cmap_swap.ttf 渲染它就会显示 flag——但你根本不必渲染，也能把 flag 还原出来。

C1cmap

找出 flag。这正是真实网站惯用的反爬手法（猫眼、起点之流）：HTML 里塞的是一串毫无 意义的码位，靠随页附带的字体的 cmap，把它们映射到画出真实字符的那些字形。于是 复制文字得到的是乱码，看却看得清清楚楚。把这张映射反过来，从字形上读出 flag。

提示 来自模块 2：font.getBestCmap() 给你一个 {码位: 字形名} 字典。 这只字体的 cmap 只含那些私用区码位——没有一条映射 ASCII。字形名倒是原封未动 （它们住在 post 表里，“只为方便人看”——模块 2），所以名叫 braceleft 的字形画的就是 {。按码位从小到大遍历，把每个字形名翻回它对应的字符。
看答案

整道题就是一次 cmap 反查——不用渲染，除了 fontTools 什么库都不用：

from fontTools.ttLib import TTFont
from fontTools.agl import toUnicode      # glyph name -> character

f = TTFont("cmap_swap.ttf")
cmap = f.getBestCmap()                 # {codepoint: glyphName}, Private-Use only
flag = "".join(toUnicode(cmap[cp]) for cp in sorted(cmap))
print(flag)

cmap 里存着 U+E000→f、U+E001→l、U+E002→a、 U+E003→g、U+E004→braceleft……按码位排序，就还原了作者安排的顺序 （这个顺序是序列唯一被记下来的地方——字形本身在文件里是按 GID 平铺的，所以单看 post 只能拿到字符的集合，拼不出 flag）。输出：

flag{c0d3po1nts_can_l13}

为什么说“cmap 会撒谎”（模块 2 的字面演绎）：cmap 是一张查找表， 不是恒等映射——U+E000 这个字节本身没有什么天生的字形，只有这张表指派给它的那一个。 更硬核的版本会把 post 名字剥掉（设成 3.0 格式），逼你去渲染每个被映射的字形、 从轮廓认字（模块 3）——同一个道理，只是难一档。

挑战 · GSUB

2 · 只在整形时才存在的 flag

这只字体的 cmap 是老实的——你把它导出来看，每个字符都映射到自己： @ 映射到 at 符号，a 映射到 a。cmap 里没有 flag，字形名里没有， strings 也一无所获。可网站上一条横幅，硬是把孤零零一个 @ 在屏幕上变成了一整个 flag。它是从哪儿冒出来的？

gsub_reveal.ttf

flag 是在整形那一步现拼出来的——就是“字符”与“最终字形”之间的那一环（模块 6）。 它只存在于渲染出的字形串里，从不以文本形式存下来。

C2GSUB

找出 flag。回忆模块 6 的流水线：cmap → GSUB（改写）→ GPOS（重定位）→ 绘制。GSUB 规则能做 cmap 物理上做不到的事——把 一个字形变成许多字形（多重替换）。这里，一条 calt lookup 把种子字形 @ 展开成整串 flag 字形。去读这条 lookup，或者干脆把种子整形一遍。

提示 两条路。静态：把布局表导出来——ttx -t GSUB gsub_reveal.ttf， 或用 fontTools 遍历它——找一条多重替换（MultipleSubst，LookupType 2）。它的 .mapping 是 {输入字形: [输出字形们]}。动态：用 HarfBuzz 整形字符串 "@"（模块 6 的 hb.shape），把结果字形名读出来。
看答案

静态——直接从 GSUB 里把替换读出来：

from fontTools.ttLib import TTFont
from fontTools.agl import toUnicode

f = TTFont("gsub_reveal.ttf")
for lookup in f["GSUB"].table.LookupList.Lookup:
    for sub in lookup.SubTable:
        if getattr(sub, "mapping", None):        # MultipleSubst (type 2)
            for src, dst in sub.mapping.items():
                print(src, "->", "".join(toUnicode(g) for g in dst))

打印出 at -> flag{gsub_1s_a_r3wr1t3r}。@ 字形被映射到了那串 24 个字形的 序列 [f, l, a, g, braceleft, g, s, u, b, …, braceright]。

动态——让整形器替你做（模块 6）：

import uharfbuzz as hb
face = hb.Face(open("gsub_reveal.ttf", "rb").read())
buf = hb.Buffer(); buf.add_str("@"); buf.guess_segment_properties()
hb.shape(hb.Font(face), buf, {"calt": True})
order = TTFont("gsub_reveal.ttf").getGlyphOrder()
print("".join(order[i.codepoint] for i in buf.glyph_infos))

一个输入 cluster 被整形成了 24 个字形。flag：

flag{gsub_1s_a_r3wr1t3r}

为什么 cmap 干不了这件事（模块 2 对比模块 6）：cmap 把每个码位映射到 恰好一个字形——它没法把一个输入扇出成一串，也没法看上下文。改写字形串是 GSUB 的活儿，发生在 cmap 之后、只在整形时进行。这正是为什么这个 flag 对 strings、对 cmap、对字形名都隐形——它压根没存在任何地方，而是被算出来的。

复现 / 加难 ~/.venv/bin/python specimens/ctf/build_ctf.py 会从样本字体重建两只字体，写文件前把每个 flag 重新验一遍（打印 PASS/PASS）。想更难？把 post 名字剥掉， 第一题就得靠渲染轮廓来破（模块 3）；把第二题那条朴素的多重替换换成链式上下文 lookup， flag 就只在某个触发串旁边才现身，而那个触发串还得你先从 ScriptList/FeatureList 里挖出来。
还有一种经典套路，你现在也有工具对付了 第三种常见套路把 flag 藏进轮廓里：cmap 看着人畜无害，可某些字形的曲线画的正是 flag 的字符，还被映射到零散的码位上，正常文字里根本不露面——你得把每个字形都导出或渲染一遍才找得到 （纯模块 3）。道理和这两道题一样：信字节，别信渲染。
